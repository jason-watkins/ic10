## 1. Introduction

IC20 is a statically typed, compiled language with Rust-inspired syntax that targets IC10 assembly for the game Stationeers. IC10 is a MIPS-inspired instruction set that runs on programmable chips inside the game, with hard constraints: 16 general-purpose registers, a 512-slot stack, a maximum of 128 lines of code, and a single runtime data type (64-bit IEEE 754 floating point).

IC20 makes programming these chips ergonomic. The programmer writes named variables, infix expressions, structured control flow, and functions; the compiler maps all of this down to register-allocated IC10 assembly that fits within the target's constraints. The language is deliberately small — there are no classes, modules, heap allocation, or strings at runtime.

Because the IC10 hardware limits programs to 128 lines, minimizing instruction count is a first-class design goal. Every language construct is designed to emit as few IC10 instructions as possible; the compiler never emits an instruction that isn't strictly required.

**Compilation examples are non-normative.** Throughout this specification, **Compilation** subsections illustrate one possible IC10 output for each construct. These examples show the logical structure of the generated code but are not prescriptive: a conforming compiler is free to emit any semantically equivalent sequence of IC10 instructions. In particular, a compiler is expected to fuse comparison expressions with their enclosing conditional branch into a single IC10 branch instruction (e.g. `bgt` instead of `sgt` + `beqz`); inline small or single-call-site functions to eliminate call overhead; rotate loops to move the exit test to the bottom edge; choose argument and result registers that fit the allocator's assignment rather than emitting `move` instructions to match a fixed register window; and apply any other transformation that reduces line count without changing observable behavior. The goal is always output equivalent to what a skilled programmer would write by hand.

The rest of this chapter provides an informal overview of the language. Later chapters describe the rules in precise detail.

### 1.1 Hello World

There is no console output in Stationeers, so the traditional "Hello, World" program instead writes a value to a device. The following program turns on a light connected to pin `d0`:

```rust
device light: d0;

fn main() {
    light.On = 1.0;
}
```

IC20 source files have the file extension `.ic20`. Assuming the program above is stored in `hello.ic20`, it can be compiled with:

```shell
ic20c hello.ic20
```

which produces `hello.ic10` containing the equivalent IC10 assembly:

```ic10
s d0 On 1
```

The `device` declaration binds the name `light` to the physical device pin `d0`. The compiler uses the pin identifier `d0` directly in all emitted instructions; no `alias` directive is produced. The `fn main()` declaration defines the program's entry point. The assignment `light.On = 1.0` writes the value `1.0` to the `On` logic field of the device, turning it on.

### 1.2 Program Structure

An IC20 program is a single source file containing a sequence of top-level items. There are three kinds of top-level items: **constant declarations**, **device declarations**, and **function declarations**. Every program must have exactly one function named `main`, which serves as the entry point.

```rust
const MAX_TEMP: f64 = 350.0;

device sensor: d0;
device heater: d1;

fn clamp_temp(temp: f64) -> f64 {
    if temp > MAX_TEMP {
        return MAX_TEMP;
    }
    return temp;
}

fn main() {
    loop {
        let temp = sensor.Temperature;
        heater.Setting = clamp_temp(temp);
        yield;
    }
}
```

Constants declared with `const` are evaluated at compile time and substituted wherever they appear. Device declarations bind names to physical pins (`d0`–`d5`) or the IC housing itself (`db`). Functions contain the program's executable logic, composed of statements and expressions described in the sections that follow.

Most IC10 programs are infinite loops that read sensors, compute, write actuators, and then `yield` to pause for one game tick. The structure above is the idiomatic IC20 pattern.

### 1.3 Types and Variables

IC20 is statically typed at compile time, but all values are represented at runtime as 64-bit IEEE 754 doubles — the only data type the IC10 machine supports. The compiler tracks three surface types and enforces their constraints with no runtime overhead:

| Type   | Description                        | Runtime Representation          |
|--------|------------------------------------|---------------------------------|
| `bool` | Truth values: `true` or `false`    | `1.0` (true) or `0.0` (false)   |
| `i53`  | 53-bit signed integers             | `f64` with zero fractional part |
| `f64`  | IEEE 754 double-precision floats   | `f64` directly                  |

Variables are introduced with `let` bindings. Type annotations are optional when the type can be inferred from the initializer:

```rust
let count = 10;            // i53 (integer literal, no decimal point)
let temp = 293.15;         // f64 (decimal point present)
let active = true;         // bool

let mut pressure: f64 = 0.0;   // explicit type, mutable
const LIMIT: i53 = 100;        // compile-time constant
```

Variables are immutable by default; reassignment requires `let mut`. There is no implicit coercion between types; mixing `i53` and `f64` in an expression requires an explicit `as` cast:

```rust
let n: i53 = 5;
let ratio: f64 = n as f64 / 2.0;
```

### 1.4 Expressions

Expressions are built from operands and operators using conventional infix notation. Operator precedence follows the familiar C/Rust ordering:

```rust
let result = a + b * 2;         // multiplication before addition
let flag = temp > 300.0 && pressure < 200.0;  // comparison then logical AND
let bits = flags & 0xFF;        // bitwise AND
```

IC20 provides built-in functions that map directly to IC10 math instructions:

```rust
let s = sin(angle);
let d = sqrt(x * x + y * y);
let t = lerp(min_temp, max_temp, ratio);
let r = rand();
```

The `select` expression acts as a ternary operator, choosing between two values based on a condition:

```rust
let mode = select(is_active, 1.0, 0.0);
```

The `hash` expression computes a CRC-32 hash of a string literal at compile time, producing the device-type or name hashes used by IC10's batch instructions:

```rust
const BATTERY: f64 = hash("StructureBattery");
```

### 1.5 Statements

The actions of a program are expressed using statements. IC20 supports variable declarations, assignments, control flow, and several IC10-specific statements.

**Declaration and assignment** introduce and modify variables:

```rust
let x = 10;
let mut y = 0.0;
y = sensor.Temperature;
```

**Selection statements** choose a path based on a boolean condition:

```rust
if temp > MAX_TEMP {
    heater.On = 0.0;
} else if temp < MIN_TEMP {
    heater.On = 1.0;
} else {
    heater.Setting = target;
}
```

**Iteration statements** repeat a block of code. `loop` creates an unconditional infinite loop; `while` loops on a condition; `for`/`in` iterates over a compile-time integer range:

```rust
loop {
    yield;
}

while pressure < 100.0 {
    pressure = sensor.Pressure;
    yield;
}

for i in 0..6 {
    // i takes the values 0, 1, 2, 3, 4, 5
}
```

**Jump statements** transfer control: `break` exits the enclosing loop, `continue` skips to the next iteration, and `return` exits a function.

**IC10-specific statements** control the chip's execution timing: `yield` pauses execution for one game tick (0.5 seconds), and `sleep(n)` pauses for `n` seconds.

### 1.6 Functions

Functions are declared with `fn`, take typed parameters, and optionally return a value:

```rust
fn average(a: f64, b: f64) -> f64 {
    return (a + b) / 2.0;
}

fn is_critical(temp: f64, limit: f64) -> bool {
    return temp > limit;
}
```

Functions without a `-> type` annotation do not return a value. The `main` function is the program's entry point and is always emitted first in the compiled output.

The compiler automatically manages the IC10 calling convention: arguments are passed in registers, `ra` (the return address register) is saved and restored on the stack for nested calls, and `jal`/`j ra` instructions implement call and return. Recursion is supported up to the 512-slot stack depth.

### 1.7 Device I/O

IC20 provides dedicated syntax for interacting with the devices connected to an IC10 chip. Devices are declared at the top level, binding a name to a physical pin:

```rust
device sensor: d0;
device valve: d1;
device self: db;      // the IC housing itself
```

Logic fields are read and written using dot notation:

```rust
let temp = sensor.Temperature;
let pressure = sensor.Pressure;
valve.On = 1.0;
valve.Setting = 50.0;
```

Slot fields are accessed via the `.slot(index)` syntax:

```rust
let occupied = sensor.slot(0).Occupied;
let item_hash = sensor.slot(1).OccupantHash;
```

Batch operations read from or write to all devices of a given type on the network:

```rust
let avg = batch_read(hash("StructureBattery"), Ratio, Average);
batch_write(hash("StructureWallLight"), On, 1.0);
```

All device reads produce `f64` values. To use a device value as a boolean, an explicit comparison is required:

```rust
let is_on: bool = valve.On != 0.0;
```

## 2. Lexical Structure

### 2.1 Programs

An IC20 ***program*** consists of a single ***source file***. A source file is an ordered sequence of Unicode characters, typically encoded as UTF-8. Source files have the file extension `.ic20`.

Conceptually, a program is compiled in three steps:

1. **Lexical analysis**, which translates a stream of characters into a stream of tokens.
2. **Syntactic analysis**, which translates the stream of tokens into an abstract syntax tree.
3. **Code generation**, which translates the syntax tree into IC10 assembly text.

### 2.2 Comments

Two forms of comments are supported: single-line comments and delimited comments.

***Single-line comments*** start with the characters `//` and extend to the end of the line:

```rust
// This is a single-line comment
let x = 10;   // inline comment
```

***Delimited comments*** start with `/*` and end with `*/`. They may span multiple lines:

```rust
/* This is a
   delimited comment */
let y = 20;
```

Comments do not nest. The character sequences `/*` and `*/` have no special meaning within a `//` comment, and the character sequences `//` and `/*` have no special meaning within a delimited comment.

Comments are not processed within string literals (e.g. inside `hash("...")` arguments).

### 2.3 White Space

White space is defined as the space character (U+0020), the horizontal tab character (U+0009), the carriage return character (U+000D), and the line feed character (U+000A).

White space may appear between any two tokens; it is required wherever adjacent tokens would otherwise merge (e.g., `leta` is one token, not `let a`). Line terminators (CR, LF, or CRLF) separate lines and terminate single-line comments, but are otherwise treated as white space.

### 2.4 Tokens

There are several kinds of tokens: identifiers, keywords, literals, operators, and punctuators. White space and comments are not tokens, though they act as separators for tokens.

When multiple lexical productions match a sequence of characters, the lexer always forms the longest possible token. For example, `>=` is lexed as a single greater-than-or-equal operator, not `>` followed by `=`.

```grammar
token:
    identifier
    keyword
    integer-literal
    float-literal
    boolean-literal
    special-float-literal
    operator-or-punctuator
```

#### 2.4.1 Identifiers

An ***identifier*** is a name used to refer to a variable, constant, function, or device.

```grammar
identifier:
    identifier-start-character  identifier-part-character*

identifier-start-character:
    letter  |  '_'

identifier-part-character:
    letter  |  decimal-digit  |  '_'

letter:
    'a'..'z'  |  'A'..'Z'

decimal-digit:
    '0'..'9'
```

Identifiers are case-sensitive. `temp` and `Temp` are distinct identifiers.

Examples of valid identifiers: `x`, `sensor_1`, `_unused`, `maxTemp`, `i`.

An identifier must not be a keyword (§2.4.2). Device logic field names (e.g., `Temperature`, `On`) are ordinary identifiers resolved contextually in device field access expressions (§8.2).

#### 2.4.2 Keywords

A ***keyword*** is a reserved sequence of characters that cannot be used as an identifier.

```grammar
keyword:  one of
    let      const    fn       if       else     loop
    while    for      in       break    continue return
    yield    sleep    device   as       mut
    bool     i53      f64
    true     false    nan      inf
```

The following words are reserved for future use. They cannot be used as identifiers, even though they have no meaning in the current version of IC20:

```grammar
enum    extern  match   pub     ref     static
struct  super   trait   type    unsafe  where
pin     is_nan
batch_read_named    batch_write_named
batch_read_slot     batch_write_slot
batch_read_slot_named  batch_write_slot_named
bit_extract         bit_insert
```

The following identifiers have special meaning when used in specific contexts (batch operations) but are not reserved and may be used as variable names:

```grammar
batch_read   batch_write   hash   select
Average      Sum           Minimum   Maximum
```

#### 2.4.3 Integer Literals

Integer literals write values of type `i53`. Four bases are supported:

```grammar
integer-literal:
    decimal-integer-literal
    hexadecimal-integer-literal
    octal-integer-literal
    binary-integer-literal

decimal-integer-literal:
    decimal-digit  ( decimal-digit | '_' )*

hexadecimal-integer-literal:
    '0x'  hex-digit  ( hex-digit | '_' )*

octal-integer-literal:
    '0o'  octal-digit  ( octal-digit | '_' )*

binary-integer-literal:
    '0b'  binary-digit  ( binary-digit | '_' )*

hex-digit:
    '0'..'9'  |  'a'..'f'  |  'A'..'F'

octal-digit:
    '0'..'7'

binary-digit:
    '0'  |  '1'
```

Underscore characters (`_`) may appear between digits for readability and are ignored. A leading or trailing underscore within the digit sequence is not permitted.

The value of an integer literal must be representable in the range −2⁵³ to 2⁵³ (the exact-integer range of an `f64`). Exceeding this range is a compile-time error. Negative integer literals are formed by applying the unary minus operator to a positive literal.

Examples:

```rust
42              // decimal
1_000_000       // decimal with separators
0xDEAD_BEEF     // hexadecimal
0o755           // octal
0b1010_0011     // binary
```

#### 2.4.4 Float Literals

Float literals write values of type `f64`. A float literal always contains a decimal point, an exponent, or both.

```grammar
float-literal:
    decimal-digits  '.'  decimal-digits  exponent-part?
    decimal-digits  exponent-part

decimal-digits:
    decimal-digit  ( decimal-digit | '_' )*

exponent-part:
    ( 'e' | 'E' )  ( '+' | '-' )?  decimal-digits
```

As with integer literals, underscore characters may appear between digits for readability.

A float literal without an exponent must contain a decimal point with digits on both sides. The bare token `1.` is not a valid float literal; write `1.0` instead.

Examples:

```rust
3.14            // simple decimal
0.5             // leading zero
1_000.0         // with separator
1.0e10          // with exponent
2.5e-3          // with negative exponent
1e5             // integer-looking but has exponent, so f64
```

#### 2.4.5 Boolean Literals

There are two boolean literals, `true` and `false`:

```grammar
boolean-literal:
    'true'
    'false'
```

The type of a boolean literal is `bool`. At runtime, `true` is represented as `1.0` and `false` as `0.0`.

#### 2.4.6 Special Float Literals

Two special literals represent IEEE 754 non-finite values:

```grammar
special-float-literal:
    'nan'
    'inf'
```

| Literal | Value               | Type  |
|---------|---------------------|-------|
| `nan`   | Quiet NaN           | `f64` |
| `inf`   | Positive infinity   | `f64` |

Negative infinity is written as the unary negation `-inf`. It is not a single literal but the unary minus operator applied to `inf`.

These literals are keywords (§2.4.2) and cannot be used as identifiers.

#### 2.4.7 Operators and Punctuators

The following tokens are operators or punctuators:

```grammar
operator-or-punctuator:  one of
    +    -    *    /    %
    &    |    ^    ~
    <<   >>
    ==   !=   <    >    <=   >=
    &&   ||   !
    =
    (    )    {    }    ;    :    ,    .    ->    ..
```

**Arithmetic operators:** `+  -  *  /  %`

**Bitwise operators:** `&  |  ^  ~`

**Shift operators:** `<<  >>`

**Comparison operators:** `==  !=  <  >  <=  >=`

**Logical operators:** `&&  ||  !`

**Assignment:** `=`

**Punctuators:** `(  )  {  }  ;  :  ,  .  ->  ..`

The `->` punctuator is used in function return type declarations. The `..` punctuator is used in `for`/`in` range expressions. The dot `.` is used in device field access and slot access expressions.

## 3. Types

### 3.1 Overview

IC20 is *statically typed at compile time* but *single-typed at runtime*. Every value in the underlying IC10 machine is a 64-bit IEEE 754 double (`f64`). The compiler tracks three surface types, enforces their constraints during type-checking, and rejects programs that violate them. Because the runtime representation is always `f64`, no conversion code is emitted for type-safe operations — the type system is zero-cost.

The three types are:

| IC20 Type | Runtime Representation          | Valid Range                                         |
|-----------|---------------------------------|-----------------------------------------------------|
| `bool`    | `0.0` (false) or `1.0` (true)   | Exactly `0` or `1`                                  |
| `i53`     | `f64` with zero fractional part | −2⁵³ to 2⁵³ (53-bit signed)                         |
| `f64`     | IEEE 754 double                 | All finite doubles, plus `nan`, `inf`, and `-inf`   |

There is **no implicit coercion** between types. Mixing `i53` and `f64` in a single expression is a compile-time error; the programmer must insert an explicit `as` cast. Similarly, using a numeric value where `bool` is expected requires an explicit comparison.

The three type names are keywords (§2.4.2) and cannot be used as identifiers:

```grammar
type:
    'bool'
    'i53'
    'f64'
```

### 3.2 Boolean (`bool`)

The `bool` type represents truth values. The only values of type `bool` are the literals `true` and `false` (§2.4.5).

```rust
let door_open: bool = false;
let is_hot: bool = temp > 300.0;   // comparison operators produce bool
```

**Logical operators.** The operators `&&` (logical AND), `||` (logical OR), and `!` (logical NOT) require operands of type `bool` and produce `bool`. Passing a non-`bool` operand is a compile-time error.

**Comparison operators.** The relational and equality operators (`==`, `!=`, `<`, `<=`, `>`, `>=`) always produce a value of type `bool`, regardless of the operand type.

**No implicit truthiness.** An `i53` or `f64` value may not appear where `bool` is expected. To test whether a numeric value is non-zero, write an explicit comparison:

```rust
let n: i53 = 5;
if n != 0 { /* … */ }     // correct
// if n { /* … */ }        // compile error: expected bool, found i53
```

**Casting from `bool`.** A `bool` may be explicitly cast to a numeric type:

```rust
let b: bool = true;
let i: i53 = b as i53;    // 1
let f: f64 = b as f64;    // 1.0
```

**Casting to `bool`.** Numeric types may **not** be cast to `bool`; the expression `n as bool` is a compile-time error. Use an explicit comparison (`n != 0` or `f != 0.0`) instead.

**IC10 lowering.** `&&` compiles to `min`, `||` compiles to `max`, and `!` compiles to `seqz`; see §5.6 for full semantics, including the lack of short-circuit evaluation.

### 3.3 Integer (`i53`)

The `i53` type represents 53-bit signed integers. All integer literals without a decimal point or exponent are of type `i53` (§2.4.3). The valid range is −2⁵³ to 2⁵³ — the set of integers representable exactly in the mantissa of an IEEE 754 double. An integer literal whose value falls outside this range is a compile-time error.

```rust
let count: i53 = 0;
let flags: i53 = 0b1010_0011;
let addr:  i53 = 0xFF;
let mask:  i53 = 0o755;
// let big: i53 = 9_007_199_254_740_993;  // compile error: out of i53 range
```

**Arithmetic operators.** `i53` supports `+`, `-`, `*`, `/`, and `%`. The division operator has a special type rule:

- **`/` — floating-point division.** When both operands are `i53`, the result type is `f64`, not `i53`. This matches the single IC10 `div` instruction, which always produces a full IEEE 754 quotient with no truncation. Emits only `div`. To obtain a truncated integer quotient, cast the result explicitly: `(a / b) as i53` emits `div` followed by `trunc`.

- **`%` — floor modulo.** Maps directly to the IC10 `mod` instruction, which computes true mathematical modulo: the result always has the same sign as the *divisor*. The result type is `i53`. This differs from C/Rust `%` (truncating remainder), where the result has the same sign as the *dividend*.

The difference between `%` and C/Rust remainder is only visible when operands have mixed signs:

```rust
let a: i53 = -7;
let b: i53 = 2;
let q: f64  = a / b;            // -3.5  (IEEE 754 division; result is f64)
let qt: i53 = (a / b) as i53;  // -3    (truncated toward zero via `as i53`)
let r: i53  = a % b;            //  1   (floor mod: result is non-negative when divisor is positive)
                                //       C/Rust would give -1

let a2: i53  = 7;
let b2: i53  = -2;
let q2: f64  = a2 / b2;            // -3.5  (IEEE 754 division; result is f64)
let q2t: i53 = (a2 / b2) as i53;  // -3    (truncated toward zero via `as i53`)
let r2: i53  = a2 % b2;            // -1   (floor mod: result is non-positive when divisor is negative)
                                   //       C/Rust would give 1
```

Because `%` uses floor modulo, its result has the same sign as the divisor; `(a / b) as i53` truncates toward zero. If you need a truncating remainder consistent with `(a / b) as i53`, compute it explicitly: `a - (a / b) as i53 * b`.

When both operands are non-negative, `(a / b) as i53` and `a % b` agree with C/Rust behavior.

**Bitwise and shift operators.** The bitwise operators (`&`, `|`, `^`, `~`) and shift operators (`<<`, `>>`) require `i53` operands and produce `i53`. Before each operation, the IC10 hardware converts the `f64` register value to a 64-bit signed integer by mathematical value (not by reinterpreting the IEEE 754 bit pattern — those representations are unrelated), and converts the result back to `f64` afterward. For `i53`-range values this round-trip is always lossless.

The right shift (`>>`) is an arithmetic shift that sign-extends, so `n >> k` equals `⌊n / 2ᵏ⌋` for signed integers. The operators `|`, `^`, `~`, and `<<` may produce results outside the `i53` range on boundary values; the compiler does not validate this.

The full operator table, individual operator semantics, and IC10 compilation are given in §5.3 (bitwise) and §5.4 (shift).

**No implicit mixing.** Combining `i53` and `f64` in the same binary expression is a type error. To widen an integer to a float, cast explicitly:

```rust
let n: i53 = 5;
let ratio: f64 = n as f64 / 2.0;
```

**IC10 lowering.** `/` compiles to `div` and `%` compiles to `mod`; see §5.2 for the full operator-to-instruction mapping. To obtain a truncated integer quotient, write `(a / b) as i53`, which compiles to `div` followed by `trunc`.

### 3.4 Float (`f64`)

The `f64` type is a full IEEE 754 double-precision floating-point number. Float literals always contain a decimal point, an exponent, or both (§2.4.4). The special keywords `nan` and `inf` (§2.4.6) are also of type `f64`.

```rust
let temp: f64 = 293.15;
let ratio: f64 = 0.5;
let tiny: f64 = 2.5e-3;
let big: f64 = 1.0e10;
```

**Arithmetic operators.** `f64` supports `+`, `-`, `*`, `/`, and `%` with the standard IEEE 754 semantics described in §5.2. The `%` operator uses floor modulo, not truncating remainder — see §3.3 for detailed sign conventions. NaN and infinity propagate through all arithmetic per IEEE 754.

**Special values.** `nan` represents a quiet NaN, `inf` represents positive infinity, and `-inf` (the unary minus operator applied to `inf`) represents negative infinity. These values behave according to standard IEEE 754 rules: all ordered comparisons involving `nan` (`==`, `<`, `>`, `<=`, `>=`) produce `false`, while `!=` produces `true` because NaN is unequal to every value including itself. Arithmetic with `inf` follows the usual IEEE 754 conventions.

**Math intrinsics.** The built-in math functions (§5.11) — including `sin`, `cos`, `sqrt`, `abs`, `min`, `max`, `lerp`, `clamp`, `rand`, and others — all accept and return `f64`. Passing an `i53` argument is a type error; cast with `x as f64` first.

### 3.5 Type Inference

IC20 uses local, unidirectional type inference. The type of an expression is always determined from the expression itself and its immediate context, with no global constraint solving.

The inference rules are:

1. A numeric literal **without** a decimal point or exponent is `i53`.
2. A numeric literal **with** a decimal point or exponent is `f64`.
3. The literals `true` and `false` are `bool`.
4. Binary arithmetic and bitwise operators require both operands to be the same type. Most return the operand type; the `/` operator is an exception — it always returns `f64` regardless of operand type, so `i53 / i53` yields `f64`. If operand types differ, the compiler reports an error at the span of the mismatched operand.
5. Comparison operators (`==`, `!=`, `<`, `<=`, `>`, `>=`) require both operands to be the same type and always produce `bool`.
6. The declared return type of a function constrains `return` expressions inside the function body. A mismatch between the return type annotation and the type of a `return` expression is a compile-time error.
7. A `let` binding takes the type of its initializer unless an explicit type annotation is present. When both an annotation and initializer are present, their types must agree.

```rust
let a = 10;            // rule 1 → i53
let b = 3.14;          // rule 2 → f64
let c = a + 20;        // rule 4 → i53 (both operands i53)
let q = a / 20;        // rule 4 (/ exception) → f64 (i53 / i53 yields f64)
let d = b > 0.0;       // rule 5 → bool
let e: f64 = 1.0;      // rule 7 → f64 (annotation matches initializer)
// let f: f64 = 10;    // compile error: rule 7 conflict (i53 ≠ f64)
```

Inference errors are reported with the source span of the conflicting expression, identifying both the expected and actual types.

### 3.6 Explicit Casts

Type conversions use the `as` keyword. Only the following casts are permitted; all others are compile-time errors. The full grammar for cast expressions is given in §5.8.

| Cast               | Behavior                                                                               |
|--------------------|----------------------------------------------------------------------------------------|
| `i53` as `f64`     | Exact widening — always lossless within the `i53` range.                               |
| `f64` as `i53`     | Truncates toward zero. Compile-time warning if the value may be fractional at runtime. |
| `bool` as `i53`    | `false` → `0`, `true` → `1`.                                                           |
| `bool` as `f64`    | `false` → `0.0`, `true` → `1.0`.                                                       |
| `i53` as `bool`    | **Compile error** — use `val != 0` instead.                                            |
| `f64` as `bool`    | **Compile error** — use `val != 0.0` instead.                                          |

Identity casts (`f64 as f64`, `i53 as i53`, `bool as bool`) are accepted but produce a compiler warning as they have no effect.

**Device I/O types.** Device register reads (e.g. `sensor.Temperature`) and slot reads always produce `f64`, because the IC10 machine makes no type distinction for device values. To obtain a `bool` from a device field, compare explicitly:

```rust
device heater: d1;

let on: bool = heater.On != 0.0;    // correct: comparison yields bool
// let on: bool = heater.On;         // compile error: f64 is not bool
```

## 4. Variables

### 4.1 Local Variables

A ***local variable*** is a named storage location declared within a function body using a `let` statement (§6.1). Every local variable must be explicitly initialized at the point of declaration.

The optional `: type` annotation declares the variable's type; if omitted, the type is inferred from the initializer expression (§3.5). When both are present, the annotation and initializer types must agree.

```rust
let x = 10;               // i53, inferred
let temp: f64 = 293.15;   // f64, annotated (matches initializer)
let active = true;        // bool, inferred
```

Local variables are not zero-initialized implicitly. Because declarations require an initializer, every path to a use of a variable passes through its declaration, making it impossible to read an uninitialized variable.

### 4.2 Mutability

By default, a `let` binding is ***immutable***: after the initializer runs, the variable cannot be reassigned.

```rust
let x = 10;
// x = 20;   // compile error: cannot assign to immutable variable `x`
```

To permit reassignment, declare the variable with `let mut`:

```rust
let mut count = 0;
count = count + 1;   // valid
```

Only `let mut` variables may appear on the left-hand side of an assignment statement (§6.2). All arguments are passed by value; passing a value to a function does not require mutability at the call site.

The `mut` modifier may not be applied to `const` declarations. Constants are always immutable by definition.

### 4.3 Constants

A ***constant*** is a compile-time value declared with `const` at the top level of a program:

```grammar
const-declaration:
    'const'  identifier  ':'  type  '='  const-expression  ';'

const-expression:
    literal
    identifier [must refer to a previously declared const]
    'hash'  '('  string-literal  ')'
    '('  const-expression  ')'
    '-'  const-expression
    '!'  const-expression
    '~'  const-expression
    const-expression  'as'  type
    const-expression  '+'   const-expression
    const-expression  '-'   const-expression
    const-expression  '*'   const-expression
    const-expression  '/'   const-expression
    const-expression  '%'   const-expression
    const-expression  '&'   const-expression
    const-expression  '|'   const-expression
    const-expression  '^'   const-expression
    const-expression  '<<'  const-expression
    const-expression  '>>'  const-expression
```

Unlike `let` declarations, constants require an explicit type annotation. The initializer must be a ***constant expression***: a compile-time-evaluable expression composed entirely of literals, other constants, and the operators `+`, `-`, `*`, `/`, `%`, unary `-`, `!`, `~`, `<<`, `>>`, `&`, `|`, `^`, and the `hash` function (§5.13).

```rust
const MAX_TEMP: f64  = 350.0;
const MIN_TEMP: f64  = -10.0;
const RANGE:    f64  = MAX_TEMP - MIN_TEMP;
const BATTERY:  f64  = hash("StructureBattery");
const LIMIT:    i53  = 0xFF & 0x7F;
const ENABLED:  bool = true;
```

Constants are substituted at every use site during compilation; they occupy no register at runtime. Constant names follow the same identifier rules as variables (§2.4.1).

A constant may not shadow another constant or a device declaration. References to undeclared constants are compile-time errors.

### 4.4 Scoping

IC20 uses ***lexical scoping***. The scope of a name is the region of the program text in which the name refers to that declaration.

**Block scope.** A local variable declared by `let` is in scope from the point immediately after its declaration statement to the closing brace of the enclosing block `{ … }`.

```rust
fn main() {
    let x = 1;
    {
        let y = 2;
        let z = x + y;   // x is visible from the outer block
    }
    // let w = y;        // compile error: `y` is not in scope here
}
```

**Shadowing.** A variable declaration in an inner block may shadow a variable with the same name from an outer block. The inner declaration creates a new, independent variable; the outer variable is unchanged and becomes visible again when the inner scope ends. The two variables may have different types.

```rust
let x: i53 = 1;
{
    let x: f64 = 2.0;   // shadows outer x; has a different type
    // x is f64 here
}
// x is i53 here again
```

Declaring a variable with the same name as an existing variable in the *same* block is permitted. The new declaration replaces the old variable from that point forward; the old variable is gone and the new one may have a different type. This allows reassigning to a differently-typed binding without requiring a new name.

**`for` loop variables.** The loop variable introduced by a `for`/`in` statement is scoped to the loop body; it has type `i53` and is implicitly immutable. See §6.8 for full details.

**Top-level scope.** Constants, device declarations, and function declarations are in scope throughout the entire program. Top-level names may not be redeclared. A local variable that shadows a top-level constant within its containing block is permitted.

**Function parameters.** Function parameters are in scope throughout the entire function body. They behave as immutable bindings; to mutate a parameter's value locally, copy it into a `let mut` variable.

```rust
fn double(n: i53) -> i53 {
    // n = n * 2;        // compile error: parameter `n` is immutable
    let mut result = n;
    result = result * 2;
    return result;
}
```

## 5. Expressions

An ***expression*** is a syntactic construct that is evaluated to produce a value of some type. Expressions may have side effects (e.g. device field reads). This chapter describes all expression forms and their types, precedence, and IC10 compilation.

***Assignment*** (`x = expr`) is a statement, not an expression (§6.2). There is no assignment expression and no comma operator.

The top-level expression production is:

```grammar
expression:
    logical-or-expression
```

The grammar of the individual expression forms is defined in §5.2–§5.13 and in §5.7 for unary/postfix/primary expressions. Constant expressions (used only in `const` initializers) are a restricted subset defined in §4.3.

### 5.1 Operator Precedence and Associativity

Operators bind according to the following table, ordered from lowest precedence (top) to highest (bottom). Within the same level, operators associate in the direction shown.

| Precedence   | Operators                                                 | Associativity          |
|--------------|-----------------------------------------------------------|------------------------|
| 1 (lowest)   | `\|\|`                                                    | Left                   |
| 2            | `&&`                                                      | Left                   |
| 3            | `==`  `!=`  `<`  `>`  `<=`  `>=`                          | Non-associative        |
| 4            | `\|`                                                      | Left                   |
| 5            | `^`                                                       | Left                   |
| 6            | `&`                                                       | Left                   |
| 7            | `<<`  `>>`                                                | Left                   |
| 8            | `+`  `-`                                                  | Left                   |
| 9            | `*`  `/`  `%`                                             | Left                   |
| 10           | `as`                                                      | Left                   |
| 11           | Unary `-`  `!`  `~`                                       | Prefix (right-to-left) |
| 12 (highest) | Function calls `f(…)`, field access `.`, grouped `(expr)` | Left                   |

Comparison operators at level 3 are ***non-associative***: the expression `a < b < c` is a compile-time error. Write `(a < b) && (b < c)` instead.

Parentheses may always be used to override precedence:

```rust
let x = (a + b) * c;
let y = (a > 0.0) && (b > 0.0);
```

### 5.2 Arithmetic Operators

The binary arithmetic operators `+`, `-`, `*`, `/`, and `%` require both operands to be the same type. Mixing `i53` and `f64` operands is a compile-time type error. The `/` operator is a special case: it always returns `f64`, even when both operands are `i53`, because the underlying IC10 `div` instruction is an IEEE 754 division with no truncation. All other arithmetic operators return the operand type.

```grammar
additive-expression:
    multiplicative-expression
    additive-expression  '+'  multiplicative-expression
    additive-expression  '-'  multiplicative-expression

multiplicative-expression:
    cast-expression
    multiplicative-expression  '*'  cast-expression
    multiplicative-expression  '/'  cast-expression
    multiplicative-expression  '%'  cast-expression
```

| Operator | `i53` operands — result type and behavior  | `f64` operands — result type and behavior   | IC10 instruction(s) |
|----------|--------------------------------------------|---------------------------------------------|---------------------|
| `+`      | `i53` — integer addition                   | `f64` — IEEE 754 addition                   | `add`               |
| `-`      | `i53` — integer subtraction                | `f64` — IEEE 754 subtraction                | `sub`               |
| `*`      | `i53` — integer multiplication             | `f64` — IEEE 754 multiplication             | `mul`               |
| `/`      | **`f64`** — IEEE 754 division              | `f64` — IEEE 754 division                   | `div`               |
| `%`      | `i53` — floor modulo (sign = divisor sign) | `f64` — floor modulo (sign = divisor sign)  | `mod`               |

The `%` operator maps directly to the IC10 `mod` instruction, which computes true mathematical modulo whose result has the same sign as the *divisor*. This differs from C/Rust `%`; see §3.3 for a detailed comparison with examples.

**Integer overflow.** The IC10 hardware performs all arithmetic in `f64`. Addition, subtraction, and multiplication on `i53` values whose magnitude exceeds 2⁵³ may silently lose precision. The compiler does not diagnose overflow in non-constant expressions.

**Division by zero.** For `f64`, IEEE 754 rules apply: `x / 0.0` produces `inf` (or `-inf` for negative `x`), and `0.0 / 0.0` produces `nan`. Because `/` always returns `f64`, the same IEEE 754 rules apply when the operands are `i53`: `i53` division by zero yields `inf` or `nan`. If the result is subsequently cast to `i53`, the behavior is undefined.

### 5.3 Bitwise Operators

The binary bitwise operators and the unary complement require operands of type `i53` and produce `i53`. They are not valid on `f64` or `bool`.

```grammar
bitwise-and-expression:
    shift-expression
    bitwise-and-expression  '&'  shift-expression

bitwise-xor-expression:
    bitwise-and-expression
    bitwise-xor-expression  '^'  bitwise-and-expression

bitwise-or-expression:
    bitwise-xor-expression
    bitwise-or-expression  '|'  bitwise-xor-expression
```

| Operator | IC10 instruction | Description                       |
|----------|------------------|-----------------------------------|
| `a & b`  | `and`            | Bitwise AND                       |
| `a \| b` | `or`             | Bitwise OR                        |
| `a ^ b`  | `xor`            | Bitwise XOR                       |
| `~a`     | `not`            | Bitwise complement: `~n = −n − 1` |

The IC10 hardware converts `f64` register values to 64-bit signed two's-complement integers before performing the operation and converts the result back to `f64` afterward. For `i53`-range values this round-trip is always lossless (see §3.3 for details).

The operators `|`, `^`, `~`, and `<<` may produce results outside the `i53` range on boundary values; the compiler does not diagnose this. The IC10 also provides a `nor` instruction (bitwise NOR) but IC20 does not expose it as a language operator; NOR can be written as `~(a | b)`.

### 5.4 Shift Operators

The shift operators `<<` (left shift) and `>>` (right shift) require both operands to be of type `i53` and produce `i53`.

```grammar
shift-expression:
    additive-expression
    shift-expression  '<<'  additive-expression
    shift-expression  '>>'  additive-expression
```

| Operator | IC10 instruction | Description                                   |
|----------|------------------|-----------------------------------------------|
| `a << b` | `sla`            | Left shift: zero-fill from right              |
| `a >> b` | `sra`            | Arithmetic right shift: sign-extend from left |

`>>` is an arithmetic right shift — it fills vacated bits from the sign bit of `a`. This means `n >> k` equals `⌊n / 2ᵏ⌋` for signed integers, preserving the sign (see §3.3).

The shift count `b` is not validated at compile time. Negative shift counts or counts ≥ 64 produce implementation-defined results at the IC10 hardware level.

### 5.5 Relational Operators

Relational and equality operators compare two values of the same type and produce `bool`.

```grammar
equality-expression:
    relational-expression
    equality-expression  '=='  relational-expression
    equality-expression  '!='  relational-expression

relational-expression:
    bitwise-or-expression
    bitwise-or-expression  '<'   bitwise-or-expression
    bitwise-or-expression  '>'   bitwise-or-expression
    bitwise-or-expression  '<='  bitwise-or-expression
    bitwise-or-expression  '>='  bitwise-or-expression
```

| Operator | Condition             | IC10 instruction |
|----------|-----------------------|------------------|
| `a == b` | Equal                 | `seq`            |
| `a != b` | Not equal             | `sne`            |
| `a < b`  | Less than             | `slt`            |
| `a > b`  | Greater than          | `sgt`            |
| `a <= b` | Less than or equal    | `sle`            |
| `a >= b` | Greater than or equal | `sge`            |

Both operands must be the same type. The ordered operators `<`, `>`, `<=`, `>=` are valid for `i53` and `f64`. The equality operators `==` and `!=` are additionally valid for `bool`.

Relational expressions are ***non-associative***: `a < b < c` is a compile-time error. Write `(a < b) && (b < c)`.

**NaN comparisons.** When either operand is `nan`, `==`, `<`, `>`, `<=`, and `>=` all produce `false`. `!=` produces `true` because NaN is unequal to every value including itself. This follows IEEE 754 ordered/unordered comparison semantics.

### 5.6 Logical Operators

The logical operators `&&`, `||`, and `!` operate on `bool` values and produce `bool`. Passing a non-`bool` operand is a compile-time error.

```grammar
logical-or-expression:
    logical-and-expression
    logical-or-expression  '||'  logical-and-expression

logical-and-expression:
    equality-expression
    logical-and-expression  '&&'  equality-expression
```

| Operator   | Meaning                | IC10 instruction |
|------------|------------------------|------------------|
| `a && b`   | Logical AND            | `min`            |
| `a \|\| b` | Logical OR             | `max`            |
| `!a`       | Logical NOT (see §5.7) | `seqz`           |

**No short-circuit evaluation.** IC20 does not guarantee short-circuit evaluation of `&&` or `||`. Both operands are always evaluated before the operator is applied, because `min` and `max` are single instructions that consume both operands simultaneously. Code that relies on the first operand guarding the second (e.g., a device read that should be skipped) must use `if` instead.

### 5.7 Unary Operators

IC20 provides three prefix unary operators. The following grammar also defines the higher-precedence `postfix-expression` and `primary-expression` forms, which provide the base cases for the expression grammar.

```grammar
primary-expression:
    literal
    identifier
    '('  expression  ')'
    select-expression
    hash-expression

literal:
    integer-literal
    float-literal
    boolean-literal
    special-float-literal

postfix-expression:
    primary-expression
    call-expression
    device-read-expression

unary-expression:
    postfix-expression
    '-'  unary-expression
    '!'  unary-expression
    '~'  unary-expression
```

| Operator | Operand | Result | Meaning                            | IC10 instruction |
|----------|---------|--------|------------------------------------|------------------|
| `-`      | `i53`   | `i53`  | Arithmetic negation                | `sub r? 0 a`     |
| `-`      | `f64`   | `f64`  | Arithmetic negation                | `sub r? 0 a`     |
| `!`      | `bool`  | `bool` | Logical NOT                        | `seqz`           |
| `~`      | `i53`   | `i53`  | Bitwise complement (`~n = −n − 1`) | `not`            |

Applying `-` to a `bool`, `!` to a numeric type, or `~` to `f64` is a compile-time error. The unary `+` operator is not supported. Unary operators are right-associative; `- -x` parses as `-(- x)`.

### 5.8 Cast Expressions

A ***cast expression*** converts a value from one type to another using the `as` keyword:

```grammar
cast-expression:
    unary-expression  ( 'as'  type )*
```

`as` associates left-to-right. The permitted casts and their compile-time and runtime behavior are defined in §3.6.

**Compilation.** Most casts produce no IC10 instruction because the runtime representation is the same `f64` register; `f64 as i53` is the only cast that emits code:

| Cast          | IC10 code emitted              |
|---------------|--------------------------------|
| `i53 as f64`  | *(none — same register value)* |
| `f64 as i53`  | `trunc r? a`                   |
| `bool as i53` | *(none — already 0 or 1)*      |
| `bool as f64` | *(none — already 0.0 or 1.0)*  |

### 5.9 Function Call Expressions

A ***function call expression*** invokes a named function with a parenthesized list of argument expressions:

```grammar
call-expression:
    identifier  '('  argument-list?  ')'

argument-list:
    expression  ( ','  expression )*
```

The function name must refer to a `fn` declaration in the same program. Built-in functions (`select`, `hash`, and the math intrinsics) use the same call syntax but are special forms, not user-defined functions (§5.11–§5.13).

Arguments are evaluated left-to-right before the function body executes. The number of arguments must exactly match the function's parameter count, and each argument's type must match the corresponding parameter's declared type; there is no implicit coercion at call sites.

```rust
fn add(a: f64, b: f64) -> f64 { return a + b; }

let result = add(1.0, 2.0);    // valid
// let bad = add(1, 2.0);      // compile error: i53 passed where f64 expected
```

The type of a call expression is the function's declared return type. Calling a void function (one with no `->` annotation) in an expression context is a compile-time error; it must be used as an expression statement (§6.3) instead.

**Compilation.** Function calls compile to IC10 `jal` with the calling convention described in §7.5.

### 5.10 Device Field Access

A ***device field access expression*** reads a logic field or slot field from a device declared with `device` (§8.1):

```grammar
device-read-expression:
    identifier  '.'  identifier
    identifier  '.'  'slot'  '('  expression  ')'  '.'  identifier
```

The first identifier must be a declared device name. The second identifier is the logic field name (e.g., `Temperature`, `On`) or slot field name (e.g., `Occupied`, `OccupantHash`). All device field reads produce a value of type `f64`. The slot index in `.slot(expr)` must be of type `i53`.

Device field *writes* (`device.Field = expr`) are assignment statements, not expressions (§6.2).

The full semantics of device reads — including field name resolution, live hardware access guarantees, and IC10 compilation — are described in §8.2 (logic fields) and §8.4 (slot access). Batch operations that address all devices of a given type on the network are described in §8.5.

### 5.11 Built-in Math Functions

IC20 provides a set of built-in functions that map directly to IC10 math and trigonometric instructions. All built-in math functions accept and return `f64`. Passing an `i53` argument is a compile-time type error; cast with `x as f64` first.

Built-in functions use the same call syntax as user-defined functions (§5.9) and may not be redefined.

| Function | Signature                | IC10 instruction | Description                                                        |
|----------|--------------------------|----------------- |--------------------------------------------------------------------|
| `abs`    | `(f64) -> f64`           | `abs`            | Absolute value                                                     |
| `ceil`   | `(f64) -> f64`           | `ceil`           | Round toward +∞                                                    |
| `floor`  | `(f64) -> f64`           | `floor`          | Round toward −∞                                                    |
| `round`  | `(f64) -> f64`           | `round`          | Round to nearest integer                                           |
| `trunc`  | `(f64) -> f64`           | `trunc`          | Round toward zero (remove fractional part)                         |
| `sqrt`   | `(f64) -> f64`           | `sqrt`           | Square root                                                        |
| `exp`    | `(f64) -> f64`           | `exp`            | Natural exponential *e*^x                                          |
| `log`    | `(f64) -> f64`           | `log`            | Natural logarithm ln(x)                                            |
| `sin`    | `(f64) -> f64`           | `sin`            | Sine (argument in radians)                                         |
| `cos`    | `(f64) -> f64`           | `cos`            | Cosine (argument in radians)                                       |
| `tan`    | `(f64) -> f64`           | `tan`            | Tangent (argument in radians)                                      |
| `asin`   | `(f64) -> f64`           | `asin`           | Arcsine; result in radians                                         |
| `acos`   | `(f64) -> f64`           | `acos`           | Arccosine; result in radians                                       |
| `atan`   | `(f64) -> f64`           | `atan`           | Arctangent; result in radians                                      |
| `atan2`  | `(f64, f64) -> f64`      | `atan2`          | `atan2(y, x)` — first argument is *y*, second is *x*               |
| `pow`    | `(f64, f64) -> f64`      | `pow`            | `pow(base, exp)` = base^exp (IEEE 754)                             |
| `min`    | `(f64, f64) -> f64`      | `min`            | Minimum of two values                                              |
| `max`    | `(f64, f64) -> f64`      | `max`            | Maximum of two values                                              |
| `lerp`   | `(f64, f64, f64) -> f64` | `lerp`           | `lerp(a, b, t)` — linear interpolation; `t` is clamped to `[0, 1]` |
| `clamp`  | `(f64, f64, f64) -> f64` | `max`, `min`     | `clamp(x, lo, hi)` — clamps `x` to `[lo, hi]`                      |
| `rand`   | `() -> f64`              | `rand`           | Uniform random number in `[0, 1)`                                  |

**Note on `lerp`.** The IC10 `lerp a b c` instruction clamps `c` to `[0, 1]` internally. `lerp(a, b, t)` in IC20 reflects this exactly.

**Note on `clamp`.** IC10 has no dedicated `clamp` instruction. `clamp(x, lo, hi)` compiles to two instructions: `min r_tmp x hi` followed by `max r? lo r_tmp`.

**Note on `min`/`max` vs. logical operators.** `&&` and `||` compile to the same `min`/`max` instructions as the built-in functions (§5.6). The built-ins operate on arbitrary `f64` values; `&&`/`||` are restricted to `bool` operands at the type level.

### 5.12 The `select` Expression

The `select` expression is a ternary conditional that chooses between two values:

```grammar
select-expression:
    'select'  '('  expression  ','  expression  ','  expression  ')'
```

`select(cond, if_true, if_false)` returns `if_true` when `cond` is `true`, and `if_false` when `cond` is `false`.

The first argument (`cond`) must be of type `bool`. The second and third arguments must be the same type, and the whole expression has that type.

```rust
let mode:    f64 = select(is_active, 1.0, 0.0);
let abs_val: i53 = select(n >= 0, n, -n);
```

**Both branches are always evaluated.** `select` is not a short-circuiting form; all three arguments are computed before the instruction executes. Use `if`/`else` statements when one branch must not be evaluated (e.g., to avoid a device write).

**Compilation.** `select(cond, x, y)` compiles directly to the IC10 `select r? cond x y` instruction, which implements `(cond != 0) ? x : y`.

### 5.13 The `hash` Expression

The `hash` expression computes a CRC-32 hash of a string literal at compile time and produces a value of type `f64`:

```grammar
hash-expression:
    'hash'  '('  string-literal  ')'

string-literal:
    '"'  string-character*  '"'

string-character:
    any Unicode character except '"' and line terminators
```

String literals may not span multiple lines. The only context in which a string literal is valid in IC20 is as the argument to `hash`; string values do not exist at runtime.

`hash(s)` produces the CRC-32 hash of the UTF-8 encoding of `s`, computed with the polynomial `0xEDB88320` (ISO 3309 / PKZIP convention, bit-reversed). The result is a non-negative integer value in the range [0, 2³²) represented as `f64`. This matches the hash algorithm used by the Stationeers game engine for `PrefabHash` and `NameHash` values.

The algorithm proceeds as follows:

1. Initialize a 32-bit register `crc` to `0xFFFFFFFF`.
2. For each byte `b` of the UTF-8 encoding of `s`:
   - `crc = crc XOR b`
   - Repeat 8 times: if the low bit of `crc` is `1`, then `crc = (crc >> 1) XOR 0xEDB88320`; otherwise `crc = crc >> 1`.
3. The result is `crc XOR 0xFFFFFFFF`, interpreted as an unsigned 32-bit integer and widened to `f64`.

For ASCII strings — the common case for Stationeers prefab names — each character corresponds to exactly one byte.

```rust
const BATTERY:    f64 = hash("StructureBattery");
const GAS_SENSOR: f64 = hash("StructureGasSensor");
const MY_SENSOR:  f64 = hash("My Sensor");
```

`hash` must always be applied to a string literal at the call site. Using `hash` with a non-literal argument is a compile-time error. `hash` is evaluated entirely at compile time and produces no IC10 instruction; it is equivalent to writing the numeric hash value directly.

## 6. Statements

A ***statement*** is a syntactic construct that is executed for its effect. Statements do not produce values. This chapter describes all statement forms, their rules, and their IC10 compilation.

```grammar
statement:
    variable-declaration-statement
    assignment-statement
    expression-statement
    block
    if-statement
    loop-statement
    while-statement
    for-statement
    break-statement
    continue-statement
    return-statement
    yield-statement
    sleep-statement
```

Statements may not appear at the top level of a program; they must occur inside a function body.

### 6.1 Variable Declaration Statements

A ***variable declaration statement*** introduces a new local variable and initializes it in a single operation.

```grammar
variable-declaration-statement:
    'let'  'mut'?  identifier  ( ':' type )?  '='  expression  ';'
```

The identifier introduces the variable into the enclosing block's scope from the point immediately following the declaration to the closing brace of that block (§4.4). The variable's type is either declared explicitly with `: type` or inferred from the initializer expression (§3.5). When both are present, they must agree. The initializer is mandatory; there are no uninitialized variable declarations.

```rust
let x = 10;              // i53, inferred
let temp: f64 = 293.15;  // f64, annotated
let active = true;       // bool, inferred
let mut count = 0;       // i53, mutable
```

A declaration without `mut` creates an ***immutable*** binding: any subsequent assignment to the variable is a compile-time error. The `mut` keyword permits reassignment via assignment statements (§6.2).

`const` declarations (§4.3) are top-level items, not statements, and cannot appear inside a function body.

**Compilation.** A variable declaration evaluates the initializer and places its result in the register allocated for the variable. No IC10 instruction models the binding itself; the register assignment is recorded in the compiler's symbol table. The initializer expression is compiled in the usual way (§5).

### 6.2 Assignment Statements

An ***assignment statement*** stores a new value into a mutable variable or writes a value to a device logic or slot field. There are three syntactic forms:

```grammar
assignment-statement:
    identifier  '='  expression  ';'
    identifier  '.'  identifier  '='  expression  ';'
    identifier  '.'  'slot'  '('  expression  ')'  '.'  identifier  '='  expression  ';'
```

**Variable assignment.** The identifier on the left-hand side must refer to a `let mut` local variable (§4.2). Assigning to an immutable `let` binding, a `const`, a function parameter, or a `for` loop variable is a compile-time error. The right-hand side expression must have exactly the same type as the variable; there is no implicit coercion.

```rust
let mut count = 0;
count = count + 1;   // valid

let x = 10;
// x = 20;           // compile error: x is not mutable
```

**Device field writes.** The remaining two assignment forms write to device logic fields and slot fields respectively. The first identifier must name a declared device (§8.1). The right-hand side expression must be of type `f64`; writing `i53` or `bool` without an explicit `as f64` cast is a compile-time error. Device writes produce no value and may not appear in expression contexts.

```rust
device heater: d1;
device dispenser: d2;

heater.On      = 1.0;
heater.Setting = 50.0;
dispenser.slot(0).Occupied = 0.0;
```

The full semantics of device field writes — including field name resolution, slot indexing, and IC10 compilation — are described in §8.3 (logic field writes) and §8.4 (slot access).

**Compilation.** A variable assignment evaluates the right-hand side and emits a `move` into the variable's register, or uses the result register directly when the allocator can arrange it. Device assignment compilation is described in §8.3 and §8.4.

### 6.3 Expression Statements

An ***expression statement*** consists of a call expression followed by a semicolon:

```grammar
expression-statement:
    call-expression  ';'
```

The primary purpose of expression statements is calling functions for their side effects. A call to a void function (one with no `->` return type annotation, §7.3) *must* appear as an expression statement — using it inside a larger expression is a compile-time error (§5.9). A call to a non-void function *may* also appear as an expression statement; in that case the returned value is discarded.

```rust
fn log_state() {
    // a void function
}

log_state();        // OK: void call as expression statement
add(1.0, 2.0);      // OK: return value of non-void function discarded
```

`batch_write` (§8.5.2) is called with function-call syntax and produces no value, so it is always used as an expression statement.

**Compilation.** An expression statement translates to the IC10 code for the underlying call. No additional instructions are emitted.

### 6.4 Block Statements

A ***block*** is a sequence of zero or more statements enclosed in braces:

```grammar
block:
    '{'  statement*  '}'
```

A block introduces a new lexical scope. Variables declared with `let` inside the block are not visible outside it (§4.4). Blocks appear as function bodies, `if` branches, and loop bodies.

An empty block `{}` is valid and has no effect.

### 6.5 The `if` Statement

The ***if statement*** conditionally executes a block based on a boolean condition:

```grammar
if-statement:
    'if'  expression  block  else-clause?

else-clause:
    'else'  block
    'else'  if-statement
```

The condition expression must be of type `bool`. A condition of type `i53` or `f64` is a compile-time error; write an explicit comparison to obtain a `bool` — `x != 0`, `f != 0.0`, etc. (§3.2).

The `else if` form is syntactic sugar for an `else` clause containing a nested `if` statement. The chain depth is unlimited.

```rust
if temp > MAX_TEMP {
    heater.On = 0.0;
} else if temp < MIN_TEMP {
    heater.On = 1.0;
} else {
    heater.Setting = target;
}
```

**Compilation.** An `if` without `else` compiles to a conditional branch that skips the body when the condition is `false`:

```ic10
# if cond { body }
<evaluate cond into r_cond>
beqz r_cond if_end
<body>
if_end:
```

An `if`/`else` inserts an unconditional jump at the end of the `if` body to skip the `else` body:

```ic10
# if cond { body_a } else { body_b }
<evaluate cond into r_cond>
beqz r_cond else_start
<body_a>
j if_end
else_start:
<body_b>
if_end:
```

Chains of `else if` generate one conditional branch per clause. The compiler does not generate a jump table; long chains compile linearly.

### 6.6 The `loop` Statement

The ***loop statement*** executes a block repeatedly and unconditionally until a `break` statement is encountered:

```grammar
loop-statement:
    'loop'  block
```

A `loop` with no reachable `break` is an infinite loop. This is the idiomatic pattern for an IC10 polling loop that runs once per game tick.

```rust
loop {
    let temp = sensor.Temperature;
    heater.On = select(temp < SETPOINT, 1.0, 0.0);
    yield;
}
```

**Compilation.** The body is surrounded by a label and an unconditional back-jump. The `loop_end` label is emitted only when the loop contains at least one reachable `break` statement:

```ic10
# loop { body }
loop_start:
<body>
j loop_start
loop_end:       # only if a break is present
```

### 6.7 The `while` Statement

The ***while statement*** evaluates a condition before each iteration and executes the body only while the condition is `true`:

```grammar
while-statement:
    'while'  expression  block
```

The condition expression must be of type `bool`. It is evaluated before the first iteration; if `false` at that point, the body never executes.

```rust
while pressure < TARGET_PRESSURE {
    pump.On = 1.0;
    yield;
}
pump.On = 0.0;
```

**Compilation.**

```ic10
# while cond { body }
while_start:
<evaluate cond into r_cond>
beqz r_cond while_end
<body>
j while_start
while_end:
```

### 6.8 The `for` Statement

The ***for statement*** iterates a loop variable over a half-open integer range:

```grammar
for-statement:
    'for'  identifier  'in'  expression  '..'  expression  block
```

Both range boundary expressions must be of type `i53` and may be any expression of that type — literals, variables, function calls, or arithmetic. The loop variable has type `i53` and is ***implicitly immutable*** — assigning to it inside the body is a compile-time error. It is scoped to the block and is not visible after the closing brace (§4.4).

The range `a..b` iterates the variable over the values `a`, `a + 1`, …, `b − 1`. The lower bound is ***inclusive*** and the upper bound is ***exclusive***. If `a ≥ b` the body is never executed. Both boundary expressions are evaluated exactly once before the loop begins; subsequent mutations to variables that contributed to either bound have no effect on the iteration count.

```rust
for i in 0..6 {
    // i takes values 0, 1, 2, 3, 4, 5
}

let start = 2;
let stop  = n + 1;    // n: i53
for i in start..stop {
    // i takes values start, start+1, …, stop-1
}
```

**Compilation.** Both boundary expressions are evaluated into registers before the loop begins. The lower bound is moved into the loop variable's register; the upper bound is held in a separate register for the duration of the loop. A counted loop with a comparison, back-jump, and increment is emitted:

```ic10
# for i in lower_expr..upper_expr { body }
<evaluate lower_expr into r_i>
<evaluate upper_expr into r_upper>
for_start:
slt r_cond r_i r_upper
beqz r_cond for_end
<body>
for_continue:
add r_i r_i 1
j for_start
for_end:
```

The `for_continue` label is the target of any `continue` statements inside the body (§6.10).

If both bounds are compile-time constants and the range is small, the compiler may unroll the loop; however, unrolling is not guaranteed and the programmer must not rely on it.

### 6.9 The `break` Statement

The ***break statement*** exits the immediately enclosing `loop`, `while`, or `for` statement:

```grammar
break-statement:
    'break'  ';'
```

A `break` that appears outside any loop construct is a compile-time error. In nested loops, `break` exits only the innermost enclosing loop; outer loops continue normally.

```rust
loop {
    if done { break; }
    yield;
}
// execution resumes here
```

**Compilation.** `break` compiles to an unconditional jump to the label immediately following the enclosing loop:

```ic10
j loop_end
```

### 6.10 The `continue` Statement

The ***continue statement*** skips the remainder of the current loop body and proceeds to the next iteration:

```grammar
continue-statement:
    'continue'  ';'
```

A `continue` that appears outside any loop construct is a compile-time error. In nested loops, `continue` affects only the innermost enclosing loop.

The point to which `continue` transfers control depends on the enclosing loop kind:

| Loop kind | `continue` transfers to           |
|-----------|-----------------------------------|
| `loop`    | The top of the loop body          |
| `while`   | The condition re-evaluation       |
| `for`     | The loop-variable increment step  |

For a `for` loop, `continue` increments the loop variable before re-checking the upper bound, so the variable is always advanced even when the body is skipped.

```rust
for i in 0..10 {
    if i % 2 == 0 { continue; }
    // reached only for odd i
}
```

**Compilation.** `continue` compiles to an unconditional jump to the appropriate label for the enclosing loop:

```ic10
j <continue_target>   # loop_start | while_start | for_continue
```

### 6.11 The `return` Statement

The ***return statement*** exits the current function, optionally supplying a return value:

```grammar
return-statement:
    'return'  expression?  ';'
```

If the enclosing function has a `-> type` annotation (§7.3), the `return` statement must include an expression whose type matches that return type. If the function has no return type annotation (a void function), the expression must be omitted. Either violation is a compile-time error.

For a non-void function, every control-flow path through the function body must terminate with a `return` statement; a path that can fall off the end of the body without returning a value is a compile-time error.

For a void function, the `return;` form is optional at the end of the body; reaching the closing brace is equivalent to an implicit `return;`.

```rust
fn restrict(x: f64, lo: f64, hi: f64) -> f64 {
    if x < lo { return lo; }
    if x > hi { return hi; }
    return x;
}

fn activate() {
    heater.On = 1.0;
    return;   // optional: function would return here anyway
}
```

**Compilation.** The return value expression, if present, is evaluated and placed in the designated return register (§7.5). The function then restores any callee-saved state (in particular, a `ra` that was pushed onto the stack for non-leaf functions) before emitting `j ra` to return to the caller. The full calling convention is described in §7.5.

### 6.12 The `yield` Statement

The ***yield statement*** suspends IC chip execution for one game tick and then resumes:

```grammar
yield-statement:
    'yield'  ';'
```

`yield` takes no arguments. It resets the IC10 line execution counter, allowing up to 128 lines to execute on the next tick. All register and stack state is preserved across the suspension.

`yield` is the standard mechanism for pacing a control loop to one iteration per game tick (approximately 0.5 seconds in Stationeers):

```rust
loop {
    let temp = sensor.Temperature;
    heater.On = select(temp < SETPOINT, 1.0, 0.0);
    yield;
}
```

**Compilation.** `yield` compiles directly to the IC10 `yield` instruction.

### 6.13 The `sleep` Statement

The ***sleep statement*** suspends IC chip execution for a specified duration in seconds:

```grammar
sleep-statement:
    'sleep'  '('  expression  ')'  ';'
```

The argument expression must be of type `f64` and represents a duration in seconds. All register and stack state is preserved across the suspension. Sleep durations are implemented at tick granularity — the chip resumes at the first tick boundary on or after the requested duration has elapsed.

```rust
sleep(2.5);    // pause for approximately 2.5 seconds
sleep(t);      // pause for t seconds (t: f64)
```

**Compilation.** `sleep(n)` compiles to the IC10 `sleep` instruction:

```ic10
sleep r_n
```

## 7. Functions

### 7.1 Function Declarations

A ***function declaration*** introduces a named, callable sequence of statements. Function declarations are top-level items; they cannot be nested inside another function or inside a block.

```grammar
function-declaration:
    'fn'  identifier  '('  parameter-list?  ')'  return-type?  block

parameter-list:
    parameter  ( ','  parameter )*
```

The identifier names the function; it must be unique among all top-level names — no two functions may share a name, and a function name may not duplicate a `const` or `device` declaration. Functions may be declared in any order; forward references are permitted.

A function's ***body*** is the block that follows the parameter list and optional return type annotation. The body introduces a new scope (§4.4); parameters are in scope throughout the entire body.

```rust
fn average(a: f64, b: f64) -> f64 {
    return (a + b) / 2.0;
}

fn is_critical(temp: f64, limit: f64) -> bool {
    return temp > limit;
}

fn activate() {
    heater.On = 1.0;
}
```

**Compilation.** Each function is emitted as an IC10 label followed by the translated instructions of its body. The label is the function's name. The `main` function is always emitted first (§7.4); the order of other functions in the output is unspecified. For every function other than `main`, every exit path through the body ends with `j ra` (§7.5). The `main` function is entered directly by the IC10 hardware and does not end with `j ra` (§7.4).

### 7.2 Parameters

A ***parameter*** is a typed, named binding that receives a copy of the corresponding argument value at each call site. Each parameter requires an explicit type annotation; parameter types are never inferred:

```grammar
parameter:
    identifier  ':'  type
```

The parameter list is a comma-separated sequence of parameters enclosed in parentheses. An empty parameter list is written `()`. Each parameter name must be distinct within the same parameter list; duplicate parameter names are a compile-time error.

Parameters receive their values by copy. There are no reference parameters, output parameters, or default values.

**Immutability.** Parameters are ***immutable*** bindings within the function body (§4.4). Assigning to a parameter, even in an `if` branch, is a compile-time error. To work with a mutable copy of a parameter's value, initialize a `let mut` local variable from the parameter:

```rust
fn restrict(x: f64, lo: f64, hi: f64) -> f64 {
    let mut v = x;
    if v < lo { v = lo; }
    if v > hi { v = hi; }
    return v;
}
```

**Scope.** Parameters are in scope throughout the entire function body. A local variable declared inside the body with the same name as a parameter shadows the parameter from that point forward (§4.4). A parameter that shares a name with a top-level constant or device declaration shadows those top-level names within the function.

**Compilation.** At the call site, argument values are placed in registers `r0`, `r1`, …, `r(n−1)` in left-to-right order before the `jal` instruction (§7.5). Each parameter is initially assigned the register that carried its argument: the first parameter corresponds to `r0`, the second to `r1`, and so on. The register allocator may subsequently move parameters and local variables to other registers as needed.

### 7.3 Return Type

A ***return type annotation*** specifies the type of value a function produces. It follows the parameter list and is preceded by `->`:

```grammar
return-type:
    '->'  type
```

A function that includes a return type annotation is a ***non-void function***. A function with no return type annotation is a ***void function***.

**Non-void functions.** Every control-flow path through the body of a non-void function must terminate with a `return` statement (§6.11) whose expression has exactly the declared return type. A path that reaches the closing brace without executing a `return` statement is a compile-time error.

```rust
fn restrict(x: f64, lo: f64, hi: f64) -> f64 {
    if x < lo { return lo; }
    if x > hi { return hi; }
    return x;
}

// compile error — some paths do not return:
// fn bad(x: f64) -> f64 {
//     if x > 0.0 { return x; }
//     // no return when x <= 0.0
// }
```

**Void functions.** A void function may omit `return` entirely; reaching the closing brace is an implicit return. An explicit `return;` with no expression is permitted and has the same effect. Placing an expression after `return` in a void function is a compile-time error.

```rust
fn activate() {
    heater.On = 1.0;
    // implicit return at closing brace
}

fn deactivate() {
    heater.On = 0.0;
    return;   // explicit but optional
}
```

The return type, when declared, must be one of the three IC20 types: `bool`, `i53`, or `f64`.

The `main` function must be a void function with no parameters and no return type annotation (§7.4). Any other declaration for `main` is a compile-time error.

### 7.4 The `main` Entry Point

Every IC20 program must contain exactly one function named `main`. A program with no `main` function, or with more than one `main`, is a compile-time error.

The `main` function must accept no parameters and declare no return type:

```rust
fn main() {
    loop {
        // ...
        yield;
    }
}
```

Any declaration of `main` with parameters or a return type annotation is a compile-time error.

When the IC chip begins execution, it starts at line 0 of the compiled IC10 output. The compiler emits `main`'s body beginning at line 0, ensuring that execution falls directly into the program logic. Other functions are emitted after `main`'s body.

The typical IC20 `main` body is an infinite `loop` containing a `yield` statement, matching the standard IC10 pattern of polling sensors, computing outputs, and then pausing for one game tick:

```rust
device sensor: d0;
device heater: d1;

const SETPOINT: f64 = 300.0;

fn main() {
    loop {
        let temp = sensor.Temperature;
        heater.On = select(temp < SETPOINT, 1.0, 0.0);
        yield;
    }
}
```

**Compilation.** The compiler emits the translated body of `main` starting at line 0. Functions other than `main` follow. The `main` body does not end with `j ra`; `main` is entered by the chip hardware, not via `jal`, so no return address is set.

Every exit point of `main` — an explicit `return;` or the implicit fall-through at `main`'s closing brace — compiles to an `hcf` (halt and catch fire) instruction. This permanently stops the chip until it is reset or power-cycled. A `loop` inside `main` that never exits never reaches the `hcf`; in that case the compiler omits it, saving one instruction line.

### 7.5 Calling Convention

IC10 provides no built-in stack frame mechanism. IC20 defines a ***software calling convention*** that the compiler implements uniformly for all function calls.

#### Argument Passing

Arguments are passed in general-purpose registers. The first argument is placed in `r0`, the second in `r1`, and so on in left-to-right order. There is no stack-based argument passing; all arguments are transferred via registers.

#### Return Value

The return value of a non-void function is placed in `r0` before the function returns. The caller reads the return value from `r0` after the `jal` completes. When a function accepts arguments and also returns a value, the return value is written into `r0` on the way out, overwriting the first argument.

#### Caller-Saved Registers

All general-purpose registers (`r0`–`r15`) are ***caller-saved***. The callee may freely modify any register. Because the caller cannot know which registers the callee will use, any live value that must survive across a call must be spilled to the IC10 stack before the `jal` is emitted and restored with a corresponding `pop` afterward.

#### Return Address Management

The IC10 `jal target` instruction stores the address of the instruction immediately following the `jal` into the special register `ra` and transfers control to `target`. The callee returns to its caller with `j ra`.

A ***leaf function*** — one that makes no calls to other IC20 functions via `jal` — simply executes `j ra` at each return point without saving or restoring `ra`:

```ic10
# leaf function — ra is not modified, no push/pop needed
leafFunc:
    <body>
    j ra
```

A ***non-leaf function*** must save `ra` before executing any `jal` instruction, because `jal` overwrites `ra` with the new return address. The compiler saves `ra` by pushing it onto the IC10 stack at the very start of the non-leaf function body, and restores it with a corresponding `pop ra` immediately before each `j ra` return:

```ic10
# non-leaf function — ra is saved and restored around the inner call
nonLeafFunc:
    push ra                              # save caller's return address
    <evaluate arguments into r0, r1, …>
    jal innerFunc                        # ra overwritten with return site
    <read return value from r0>
    pop ra                               # restore caller's return address
    j ra                                 # return to caller
```

#### Full Example

```rust
fn add(a: f64, b: f64) -> f64 {
    return a + b;
}

fn scale(x: f64, factor: f64) -> f64 {
    return add(x, x) * factor;
}
```

The compiler may emit IC10 similar to:

```ic10
# fn add(a: f64, b: f64) -> f64
add:
    add r0 r0 r1    # r0 = a + b  (result in r0 per convention)
    j ra            # leaf function: return

# fn scale(x: f64, factor: f64) -> f64
scale:
    push ra         # non-leaf: save caller's return address
                    # x is in r0, factor is in r1
    move r2 r1      # save factor before r1 is overwritten for the add call
    move r1 r0      # second argument to add(x, x): b = x
                    # first argument r0 already holds x
    jal add         # r0 = add(x, x) = 2*x; ra overwritten
    mul r0 r0 r2    # r0 = 2*x * factor
    pop ra          # restore caller's return address
    j ra            # return

main:
    # ...
```

The exact register assignments and instruction sequence are produced by the register allocator and may differ from the example above.

### 7.6 Recursion

IC20 supports ***recursion***: a function may call itself directly or through a chain of calls to other functions. No special syntax or annotation is required; the standard calling convention (§7.5) handles recursive calls identically to calls between distinct functions.

Each invocation of a non-leaf function pushes `ra` onto the IC10 stack before making any further call. Because `ra` is a single `f64` value occupying one stack slot, stack consumption per recursive frame is at least 1 slot. A function that also spills live registers to the stack uses additional slots per frame.

With 512 stack slots available and one slot consumed per frame, the maximum recursion depth for a function that spills only `ra` is 512. Exceeding the IC10 stack limit produces undefined behavior at the hardware level.

```rust
fn factorial(n: i53) -> f64 {
    if n <= 1 { return 1.0; }
    return (n as f64) * factorial(n - 1);
}
```

***Mutual recursion*** is supported: function A may call function B and B may call A. Because forward references are permitted (§7.1), mutually recursive functions may be declared in any order:

```rust
fn is_even(n: i53) -> bool {
    if n == 0 { return true; }
    return is_odd(n - 1);
}

fn is_odd(n: i53) -> bool {
    if n == 0 { return false; }
    return is_even(n - 1);
}
```

**Practical considerations.** Recursive programs on IC10 chips have limited stack space. Programs that require large recursion depth should consider replacing recursion with an iterative `while` or `for` loop, which consumes no stack space beyond any locally spilled variables.

## 8. Device I/O

IC20 provides dedicated syntax for reading and writing the devices connected to an IC10 chip. Devices are declared by name at the top level of a program and are then accessed by that name using dot notation. The compiler translates device I/O expressions and statements directly to IC10 `l`, `s`, `ls`, `ss`, `lb`, and `sb` instructions.

### 8.1 Device Declarations

A ***device declaration*** binds a name to a physical device pin, making that device accessible by name throughout the program.

```grammar
device-declaration:
    'device'  identifier  ':'  device-pin  ';'

device-pin:
    'd0'  |  'd1'  |  'd2'  |  'd3'  |  'd4'  |  'd5'  |  'db'
```

Device declarations are top-level items; they cannot appear inside a function body or block. The identifier introduces the device name into the top-level scope. It must be unique among all top-level declarations: a device name may not duplicate another `device`, `const`, or `fn` declaration in the same program.

The device pin specifies the physical connector to which the device is attached:

| Pin       | Description                                                                 |
|-----------|-----------------------------------------------------------------------------|
| `d0`–`d5` | The six screwdriver-configurable cable connectors on the IC Housing         |
| `db`      | The device the IC chip is physically mounted in (the hosting device itself) |

```rust
device sensor:  d0;
device heater:  d1;
device pump:    d2;
device self:    db;    // the IC Housing itself
```

The same physical pin may not be bound to two different names; declaring two devices for the same pin is a compile-time error. Device declarations may appear in any order relative to other top-level items.

A device pin that has no physical device connected at runtime is treated by the IC10 hardware as returning `0` for any read and silently discarding any write.

**Compilation.** Device declarations produce no IC10 output. Wherever a device name appears in source code, the compiler substitutes the corresponding hardware pin identifier (`d0`–`d5`, or `db`) directly into every emitted `l`, `s`, `ls`, `ss`, `lb`, and `sb` instruction. Emitting no `alias` directives saves one instruction line per device, preserving instruction budget for program logic.

In all compilation templates in §8.2–§8.5 and §5.10, the placeholder `device` stands for the substituted pin identifier.

### 8.2 Reading Logic Fields

A ***logic field read*** retrieves the current value of a named logic property from a declared device. Logic field reads are expressions of type `f64` (§5.10).

```grammar
logic-field-read-expression:
    identifier  '.'  identifier
```

The first identifier must name a declared device (§8.1). The second identifier is the ***logic field name*** — a name identifying a logical property of the device, such as `Temperature`, `Pressure`, `On`, `Setting`, or `Ratio`. Logic field names are not keywords; they are unquoted identifiers resolved contextually and passed verbatim to the IC10 assembler. The set of valid logic field names is device-dependent and determined by the Stationeers game engine; the compiler does not validate that a given field name is supported by a particular device type.

All logic field reads produce a value of type `f64`. There is no implicit conversion to `bool` or `i53`; to obtain a `bool` from a device value, write an explicit comparison (§3.2, §3.6):

```rust
device sensor:  d0;
device valve:   d1;

let temp:     f64  = sensor.Temperature;      // f64 directly
let pressure: f64  = sensor.Pressure;         // f64 directly
let is_open:  bool = valve.Open != 0.0;       // compare to obtain bool
let mode:     i53  = valve.Mode as i53;       // cast to obtain i53
```

Device reads always perform a live hardware access. The compiler never caches or hoists device reads; each source occurrence of `device.Field` compiles to exactly one IC10 `l` instruction regardless of how many times the same expression appears.

**Compilation.** `device.Field` compiles to:

```ic10
l r_result device Field
```

### 8.3 Writing Logic Fields

A ***logic field write*** stores a value into a named logic property of a declared device. It is an assignment statement (§6.2).

```grammar
logic-field-write-statement:
    identifier  '.'  identifier  '='  expression  ';'
```

The first identifier must name a declared device (§8.1). The second identifier is the logic field name, subject to the same rules as in §8.2. The right-hand side expression must be of type `f64`; writing a value of type `i53` or `bool` without an explicit `as f64` cast is a compile-time error.

```rust
device heater: d1;
device light:  d2;

heater.On      = 1.0;
heater.Setting = 50.0;
light.Color    = 4.0;        // Red
// light.On    = 1;          // compile error: i53 is not f64
// light.On    = true;       // compile error: bool is not f64
light.On       = 1 as f64;   // explicit cast: valid
```

Device writes produce no value and may not appear in expression contexts.

**Compilation.** `device.Field = expr` compiles to:

```ic10
<evaluate expr into r_val>
s device Field r_val
```

### 8.4 Slot Access

A ***slot field read*** retrieves the current value of a named property from a numbered slot of a device. A ***slot field write*** stores a value into a numbered slot property.

Devices have one or more numbered slots (zero-indexed). A slot typically corresponds to an import port (slot 0), an export port (slot 1), or an internal bay (slot 2), though the exact layout varies by device type.

```grammar
slot-read-expression:
    identifier  '.'  'slot'  '('  expression  ')'  '.'  identifier

slot-write-statement:
    identifier  '.'  'slot'  '('  expression  ')'  '.'  identifier  '='  expression  ';'
```

The first identifier must name a declared device (§8.1). The expression inside `slot(…)` is the ***slot index*** and must be of type `i53`; it may be any expression of that type, including variables and arithmetic. The final identifier is the ***slot field name*** (e.g., `Occupied`, `OccupantHash`, `Quantity`, `Charge`, `ChargeRatio`). Slot field names are passed verbatim to the IC10 assembler; the compiler does not validate that a given slot field name is supported by a particular device or slot type.

Slot field reads produce a value of type `f64`. The right-hand side of a slot field write must also be of type `f64`.

```rust
device fabricator: d0;

let occupied:   f64 = fabricator.slot(0).Occupied;      // 1.0 if occupied
let item_hash:  f64 = fabricator.slot(0).OccupantHash;  // CRC-32 hash of item
let count:      f64 = fabricator.slot(1).Quantity;      // item quantity

fabricator.slot(2).Lock = 1.0;   // lock the internal bay slot

let slot_index: i53 = 0;
let occ = fabricator.slot(slot_index).Occupied;   // non-constant slot index
```

**Compilation.** `device.slot(index).Field` reads compile to:

```ic10
<evaluate index into r_index>
ls r_result device r_index Field
```

`device.slot(index).Field = expr` writes compile to:

```ic10
<evaluate index into r_index>
<evaluate expr into r_val>
ss device r_index Field r_val
```

### 8.5 Batch Operations

***Batch operations*** read from or write to every device of a specified type on the cable network, rather than a single directly-connected device. They identify the target device type by its ***device type hash*** — the CRC-32 of the device's prefab name (§5.13) — allowing a single instruction to affect an arbitrary number of devices without individual device declarations.

IC20 provides two batch operations: `batch_read` (an expression) and `batch_write` (a statement). Their names are contextual identifiers (§2.4.2), not keywords.

Both operations take the device type hash as their first argument and an ***unquoted logic type identifier*** as their second argument. The logic type identifier names the device property to read or write; it is not an expression and may not be a variable. It is passed verbatim to the IC10 assembler.

Device type hashes are most commonly expressed as `hash(…)` calls in `const` declarations (§9.1, §9.2):

```rust
const BATTERY_HASH: f64 = hash("StructureBattery");
const LIGHT_HASH:   f64 = hash("StructureWallLight");
```

#### 8.5.1 Batch Read

`batch_read` reads a logic property from every network device of a given type and combines the values according to a batch aggregation mode.

```grammar
batch-read-expression:
    'batch_read'  '('  expression  ','  identifier  ','  batch-mode  ')'

batch-mode:
    'Average'  |  'Sum'  |  'Minimum'  |  'Maximum'
```

`batch_read(type_hash, LogicType, mode)` searches the cable network for all devices whose `PrefabHash` matches `type_hash`, reads the value of `LogicType` from each, and aggregates the results according to `mode` (§8.5.3). The result type is `f64`. If no matching devices are found, a mode-specific default value is returned (§8.5.3).

`batch_read` may appear anywhere an `f64` expression is valid:

```rust
const BATTERY: f64 = hash("StructureBattery");
const SENSOR:  f64 = hash("StructureGasSensor");

let avg_charge:  f64 = batch_read(BATTERY, Charge,      Average);
let total_moles: f64 = batch_read(SENSOR,  TotalMoles,  Sum);
let min_ratio:   f64 = batch_read(BATTERY, Ratio,       Minimum);
let max_temp:    f64 = batch_read(SENSOR,  Temperature, Maximum);
```

**Compilation.** `batch_read(hash_expr, Field, Mode)` compiles to:

```ic10
<evaluate hash_expr into r_hash>
lb r_result r_hash Field Mode
```

#### 8.5.2 Batch Write

`batch_write` stores a value into a logic property of every network device of a given type.

```grammar
batch-write-statement:
    'batch_write'  '('  expression  ','  identifier  ','  expression  ')'  ';'
```

`batch_write(type_hash, LogicType, value)` searches the cable network for all devices whose `PrefabHash` matches `type_hash` and stores `value` into `LogicType` on each. The first argument must be of type `f64`. The value argument must be of type `f64`; passing `i53` or `bool` without an explicit `as f64` cast is a compile-time error. `batch_write` is a void operation and must be used as an expression statement (§6.3); it may not appear in expression contexts.

```rust
const LIGHT:  f64 = hash("StructureWallLight");
const SORTER: f64 = hash("StructureSorterLogic");

batch_write(LIGHT,  On,    1.0);   // turn on all wall lights
batch_write(LIGHT,  Color, 2.0);   // set all wall lights to Green
batch_write(SORTER, Mode,  1.0);
// batch_write(LIGHT, On, 1);       // compile error: i53 is not f64
```

If no devices on the network match `type_hash`, `batch_write` has no effect and produces no error.

**Compilation.** `batch_write(hash_expr, Field, val_expr)` compiles to:

```ic10
<evaluate hash_expr into r_hash>
<evaluate val_expr into r_val>
sb r_hash Field r_val
```

#### 8.5.3 Batch Modes

The batch mode argument to `batch_read` controls how values from multiple matching devices are aggregated. The four mode identifiers (`Average`, `Sum`, `Minimum`, `Maximum`) are contextual identifiers (§2.4.2) that may be used as ordinary identifiers elsewhere in the program.

| Mode      | IC10 value | No-match result | Description                                   |
|-----------|------------|-----------------|-----------------------------------------------|
| `Average` | `0`        | `nan`           | Arithmetic mean of all matching device values |
| `Sum`     | `1`        | `0`             | Sum of all matching device values             |
| `Minimum` | `2`        | `0`             | Smallest value among all matching devices     |
| `Maximum` | `3`        | `-inf`          | Largest value among all matching devices      |

The ***no-match result*** is the value returned when no device on the network has a `PrefabHash` equal to the specified hash. For `Average` this is `nan`; for `Sum` and `Minimum` it is `0`; for `Maximum` it is `-inf`.

**Testing for the `Average` no-match case.** Because `nan` compares unequal to every value including itself (§3.4), the standard IEEE 754 self-comparison idiom detects whether an `Average` result is valid:

```rust
let avg = batch_read(hash("StructureBattery"), Ratio, Average);
if avg == avg {
    // at least one battery on the network; avg is a valid mean
} else {
    // no batteries; avg is nan
}
```

`avg == avg` evaluates to `false` if and only if `avg` is `nan`.

## 9. Compile-time Features

IC20 provides three compile-time mechanisms that allow the compiler to eliminate computation from the generated IC10 output: named constants (§4.3), the `hash` function (§5.13), and general constant folding. This chapter describes their semantics, evaluation rules, and compilation.

### 9.1 Constants

The syntax, scoping, and type rules for constant declarations are defined in §4.3. This section describes their compile-time evaluation and compilation.

**Evaluation order.** All constant declarations are evaluated before any function body is compiled. Evaluation proceeds in dependency order: a constant whose initializer references another constant is evaluated after the constant it references. Because cyclic references between constants are prohibited (§4.3), a total evaluation order always exists.

**Compile-time substitution.** Every use of a constant name is replaced with its evaluated value before IC10 code is generated; the declaration itself produces no output. The substituted value is a bare `f64` number — at the IC10 level, all values are `f64` and type annotations cease to exist. No `alias` or `define` directive is emitted; the numeric value is inlined directly at each use site.

```rust
// IC20 source:
const SETPOINT: f64 = 300.0;
if temp > SETPOINT { heater.On = 0.0; }

// Equivalent after constant substitution:
if temp > 300.0 { heater.On = 0.0; }
```

### 9.2 The `hash` Function

The syntax, algorithm, and type of `hash` expressions are defined in §5.13. This section describes how `hash` participates in the compile-time evaluation model.

`hash(s)` is evaluated entirely at compile time and behaves identically to a numeric `f64` literal. It generates no IC10 instruction and participates in constant expressions (§4.3) and constant folding (§9.3). `hash` is not a callable function at runtime; it cannot be passed as a value, stored in a variable, or invoked indirectly.

The idiomatic use of `hash` is to declare named constants for the device types used by batch operations (§8.5):

```rust
const BATTERY:    f64 = hash("StructureBattery");
const GAS_SENSOR: f64 = hash("StructureGasSensor");

let avg_charge = batch_read(BATTERY, Ratio, Average);
```

`hash` may also appear inline in an expression rather than in a `const` declaration, but the constant form is preferred for readability.

### 9.3 Constant Folding

***Constant folding*** is the compile-time evaluation of expressions whose operands are all known statically. IC20 performs constant folding in two distinct contexts.

**Mandatory folding: constant declarations.** The initializer of a `const` declaration is a constant expression (§4.3) and is always evaluated entirely at compile time. A constant initializer that cannot be evaluated at compile time — for example, because it references a non-constant variable or calls a user-defined function — is a compile-time error, not a deferred computation.

**Optional folding: function bodies.** Inside function bodies, the compiler *may* evaluate a sub-expression at compile time when all of its operands reduce to compile-time-known values (constants, literals, or previously folded sub-expressions). This optimization is not guaranteed; programs must not depend on whether a given sub-expression is folded.

**Arithmetic and bitwise folding.** The four arithmetic operators (`+`, `-`, `*`, `/`), `%`, the bitwise operators (`&`, `|`, `^`, `~`), the shift operators (`<<`, `>>`), and unary negation are all folded when their operands are constant. The result obeys the same type rules as the corresponding runtime expression:

```rust
const A: i53 = 100;
const B: i53 = 7;
const C: i53 = A * B;           // 700
const Q: f64 = A / B;           // approximately 14.286 (i53 / i53 yields f64 at compile time too)
const D: i53 = 0xFF & 0x0F;     // 15
const E: f64 = 3.0 * 1.5;       // 4.5
const F: f64 = 1.0e3 + 2.5e2;   // 1250.0
```

**Cast folding.** `as` casts in constant expressions are evaluated at compile time using the same conversion rules as at runtime (§3.6). An `f64 as i53` cast in a constant expression truncates toward zero; the compiler emits a warning if the value has a non-zero fractional part.

```rust
const X: f64  = 7.9;
const Y: i53  = X as i53;   // 7 (truncated); compiler warning: fractional part discarded
const W: f64  = 8.0;
const Z: i53  = W as i53;   // 8; no fractional-part warning (exact integer)
```

**`hash` in constant expressions.** `hash(s)` is treated as a numeric `f64` constant during folding. Arithmetic built on `hash` results participates in further folding:

```rust
const H: f64 = hash("StructureBattery");   // folded to a numeric value
const M: f64 = H + 1.0;                    // folded further (unusual but valid)
```

**Division by zero.** The behavior of division and modulo by a constant zero is consistent with the runtime behavior described in §5.2:

- For `/` (both `f64` and `i53` operands), division by zero is *not* a compile-time error. Because `/` always produces `f64`, IEEE 754 rules apply in both cases: `1.0 / 0.0` and `1 / 0` both fold to `inf`, and `0.0 / 0.0` and `0 / 0` fold to `nan`. The resulting `f64` constant may hold `nan` or `inf`.
- For `%` with `i53` operands, modulo by `0` is a ***compile-time error***. The IC10 `mod` instruction by zero produces undefined behavior; the compiler rejects this statically in constant context.
- For `%` with `f64` operands, modulo by `0.0` is not a compile-time error; the result is `nan`.

```rust
const INF:  f64 = 1.0 / 0.0;        // valid: folds to inf
const INF2: f64 = 10 / 0;            // valid: i53/i53→f64, folds to inf
const NAN:  f64 = 0.0 / 0.0;        // valid: folds to nan
// const BAD: i53 = 10 % 0;          // compile error: i53 modulo by zero
```

**NaN and infinity.** The literals `nan` and `inf` are valid `f64` constant operands. Arithmetic involving them follows IEEE 754 semantics during folding; for example, `nan + 1.0` folds to `nan`, and `inf - inf` folds to `nan`.

**Out-of-range `i53` results.** If constant folding produces an `i53` result outside the range −2⁵³ to 2⁵³, the compiler reports a compile-time error. For `f64` constants, all finite results and the special values `nan`, `inf`, and `-inf` are representable and valid.

## 10. Execution Model

### 10.1 Program Lifecycle

An IC20 program compiles to a single `.ic10` assembly file. Execution of this file is driven by the IC10 chip hardware inside the Stationeers simulation.

**Startup.** When the IC chip receives power after being unpowered, or when the chip is reset by external means, all general-purpose registers (`r0`–`r15`) are reset to `0.0` and the stack pointer (`sp`) is reset to `0`, discarding any previously saved stack values. The program counter is set to line 0, and execution begins at the first instruction of `main`'s body. No explicit jump is needed; control falls through sequentially.

**Normal operation.** The intended lifecycle for most IC20 programs is an infinite polling loop:

```rust
fn main() {
    loop {
        // read sensors, compute, write actuators
        yield;
    }
}
```

On each game tick, the loop body executes once, reads the current state of connected devices, performs computation, writes outputs, and then `yield`s to pause until the next tick. This pattern matches the standard IC10 programming idiom.

**Execution resumes after `yield` and `sleep`.** After `yield` (§6.12) or `sleep` (§6.13), the chip resumes at the instruction immediately following the pause point. All register and stack state is preserved across the suspension. The program counter does not reset to line 0 on resume; it continues from where it left off.

**Termination.** Every exit from `main` — an explicit `return;` or the implicit fall-through at `main`'s closing brace — causes the compiler to emit an `hcf` (halt and catch fire) instruction (§7.4). This permanently stops the chip until it is externally reset or power-cycled. When `main` runs an infinite `loop`, the `hcf` is never reached and is omitted by the compiler.

**Power loss.** If the chip loses power, execution stops and all register and stack state is discarded. When power is restored, execution restarts from line 0 as described under Startup above.

### 10.2 Tick Model

IC10 programs run inside the Stationeers simulation, which advances in discrete ***game ticks***. One game tick corresponds to 0.5 seconds of simulation time.

**Tick execution budget.** On each tick, an IC chip may execute at most 128 lines of its program. Every IC10 instruction occupies one line. Empty lines count toward the 128-line limit as well.

**Automatic pause.** If 128 lines execute within a single tick without an explicit `yield` or `sleep`, the chip ***automatically pauses*** for the remainder of the tick and resumes at the following instruction on the next tick. This automatic pause is transparent: no registers or stack state are affected; the IC10 counter simply resets for the next tick. A long computation that spans many lines may therefore be silently spread across multiple ticks by the auto-pause mechanism.

**The `yield` statement.** `yield` (§6.12) explicitly pauses the chip for one tick. The chip resumes at the instruction following the `yield` on the next tick, with a fresh 128-line budget. Using `yield` at the end of the main polling loop ensures the loop runs at most once per tick.

**The `sleep` statement.** `sleep(n)` (§6.13) pauses the chip for `n` seconds. The duration is measured in ticks; the chip resumes at the first tick boundary on or after the requested interval has elapsed.

**Instruction budget considerations.** The total number of IC10 lines a program occupies is bounded by 128 (§10.3). The number of those lines that execute in a single tick depends on the program's control flow. A `main` loop containing a `yield` runs at most 128 lines — the entire program — per tick before the explicit `yield` resets the budget. Programs that perform expensive computation (large `for` loops, many function calls, deep recursion) may exhaust the tick budget before reaching `yield`, causing the game state written in that tick to be incomplete. Keeping the main loop body significantly below 128 lines leaves a safety margin.

### 10.3 IC10 Constraints

The IC10 hardware imposes hard limits that every compiled IC20 program must satisfy. The compiler enforces or diagnoses violations wherever they can be determined statically.

**Program length — 128 lines.** The compiled IC10 output may contain at most 128 lines, numbered 0–127. Each instruction occupies exactly one line. If the compiler determines that the compiled output exceeds 128 lines, it reports a compile-time error and does not produce output. Because line count depends on the number of functions, their complexity, and how many registers the allocator must spill to the stack, programs near the limit may succeed or fail depending on minor code changes.

Approximate line costs for common constructs:

- Device field write with a simple expression: 2–3 lines
- `if`/`else`: 2 lines overhead plus the bodies
- `while` loop: 2 lines overhead plus the body
- Function call: 1 line (`jal`) plus argument setup (1 line per argument)
- Non-leaf `ra` save and restore: 2 lines (`push ra`, `pop ra`)
- Stack spill and restore per variable: 2 lines (`push`, `pop`)

**General-purpose registers — 16.** The chip has 16 general-purpose registers (`r0`–`r15`). When more than 16 variables are simultaneously live within a function — including temporaries holding intermediate expression values — the register allocator must ***spill*** the excess onto the IC10 stack, emitting `push` and `pop` instructions. Spills increase both line count and stack usage. The programmer can reduce register pressure by keeping functions short, limiting the number of live variables, and introducing intermediate `let` bindings that give the allocator more flexibility.

**Stack depth — 512 slots.** The IC10 stack can hold at most 512 `f64` values. Stack slots are consumed by:

- `ra` saves for non-leaf functions (§7.5): one slot per active non-leaf call frame.
- Register spills: one slot per spilled register per call frame.
- Recursive calls: each recursive invocation of a non-leaf function contributes the above costs again.

Exceeding the stack limit produces undefined hardware behavior; there is no stack-overflow exception. The compiler does not statically bound recursive stack usage; the programmer is responsible for ensuring that the maximum recursion depth multiplied by the per-frame stack cost does not exceed 512.

**Device connections — 7.** The IC Housing provides seven device connections: pins `d0`–`d5` (six cable connectors) and `db` (the device the chip is mounted in). A program may declare at most one device per pin, and the set of valid pin identifiers is exactly `{d0, d1, d2, d3, d4, d5, db}` — the grammar already enforces this (§8.1). If a pin appears in two `device` declarations, the compiler reports a duplicate-pin error.

**Single value type — `f64`.** Every IC10 register holds a 64-bit IEEE 754 double. IC20's three surface types (`bool`, `i53`, `f64`) share this single runtime representation. Type errors are always caught at compile time; no runtime type tag exists and no type mismatch can occur at runtime. The only cast that emits IC10 code is `f64 as i53`, which emits a `trunc` instruction; all other casts are zero-cost (§5.8).

**Summary of hardware constraints.**

| Resource              | Limit | Compiler action when exceeded      |
|-----------------------|-------|------------------------------------|
| Program lines         | 128   | Compile-time error                 |
| General registers     | 16    | Spill excess to stack              |
| Stack slots           | 512   | No static check; runtime undefined |
| Device pins           | 7     | Compile-time error (duplicate pin) |
| Value type at runtime | `f64` | N/A — always `f64`                 |
