# IC10 Machine Architecture & Assembly Language Specification

*Source: [Stationeers Wiki](https://stationeers-wiki.com/IC10) — MIPS-inspired scripting language for the game Stationeers.*

---

## 1. Execution Model

- Programs run on **IC10 chips** inserted into IC Housings (or directly into some devices like Air Conditioners).
- A program is a list of up to **128 lines** (lines 0–127).
- The IC executes **128 lines per tick** (game tick = 0.5 seconds). Empty lines count toward this limit.
- After 128 lines, execution automatically pauses for 1 tick and resumes where it left off.
- `yield` explicitly pauses for 1 tick; `sleep a` pauses for `a` seconds.
- `hcf` (halt and catch fire) stops execution permanently.
- Most programs are infinite loops ending with a jump back to the start.

## 2. Registers

### 2.1 General-Purpose Registers (r0–r15)

16 registers named `r0` through `r15`. All computation is done through these registers.

### 2.2 Special Registers

| Register | Purpose                                                                                                                           |
|----------|-----------------------------------------------------------------------------------------------------------------------------------|
| `ra`     | **Return address.** Set by `-al` suffixed branch/jump instructions (e.g., `jal`, `beqal`). Used for subroutine return via `j ra`. |
| `sp`     | **Stack pointer.** Tracks the next index for `push`/`pop`. Range: 0–512. Readable and writable like any register.                 |

Neither `ra` nor `sp` is protected — any instruction can modify them.

### 2.3 Device Registers (d0–d5, db)

| Register  | Purpose                                                                                            |
|-----------|----------------------------------------------------------------------------------------------------|
| `d0`–`d5` | Devices connected via screwdriver-configurable pins on the IC Housing.                             |
| `db`      | The device the IC chip is physically mounted in (the housing itself, or an Air Conditioner, etc.). |

Device registers are not directly readable — use `l` (load) and `s` (set) instructions with a **LogicType** (e.g., `Temperature`, `Pressure`, `On`).

## 3. Data Types & Numeric Literals

- All values are **double-precision floating point** (64-bit IEEE 754). *(Wiki notes this is "confirmed?"—treat with slight caution.)*
- No distinct integer type, but doubles can represent integers exactly up to ~2^53.
- **Bitwise operations** internally treat values as **64-bit signed two's complement integers** (range: −2^63 to 2^63−1).

### Literal Formats

| Prefix   | Base        | Example                                                         |
|----------|-------------|-----------------------------------------------------------------|
| *(none)* | Decimal     | `123.456`, `-10`                                                |
| `$`      | Hexadecimal | `$E1B2`, `$DEADBEEF`                                            |
| `%`      | Binary      | `%01101001` (underscores ignored for readability: `%0110_1001`) |

### Built-in Constants

| Name   | Value                    |
|--------|--------------------------|
| `pinf` | Positive infinity        |
| `ninf` | Negative infinity        |
| `nan`  | Not a Number (quiet NaN) |

`HASH("string")` is a compile-time macro that computes the CRC-32 hash of a string. Used for batch device operations and reagent lookups.

`LogicType.X` syntax (e.g., `LogicType.Channel0`) resolves to the integer enum value of that LogicType.

### Operand Type Notation

The instruction reference tables use this notation to indicate what each operand accepts:

| Notation | Meaning |
| --- | --- |
| `r?` | Any general-purpose register (`r0`–`r15`, `ra`, `sp`) — write destination or register-only source |
| `a(r?\|num)` | Named input operand accepting either a register **or** a numeric literal (decimal, hex `$…`, binary `%…`, `define` constants, `HASH(…)`, `LogicType.*`, `pinf`, `ninf`, `nan`) |
| `d?` | A device register (`d0`–`d5`, `db`) or an alias for one |
| `(d?\|dr?)` | A device operand that additionally accepts indirect access: `dr?` uses the value in register `r?` as the device index (0–5) |

The destination register is always written as `r?` (never `(r?\|num)`). Every other operand that shows `(r?\|num)` accepts either a register or a literal.

## 4. Addressing Modes

### Direct

```ic10
move r0 42        # literal → register
add r1 r0 3       # register + literal → register
l r0 d0 Temperature  # device logicType → register
```

### Indirect Register (`rr?`)

Prefixing `r` before a register dereferences it as a pointer (value must be 0–15):

```ic10
move r0 5
move rr0 10       # equivalent to: move r5 10
```

Multiple levels of indirection are supported:

```ic10
move r1 2
move r2 3
move rrr1 4       # r1→2, r2→3, so rrr1 = r3; equivalent to: move r3 4
```

### Indirect Device (`dr?`)

```ic10
move r0 2
s dr0 On 1        # equivalent to: s d2 On 1
```

### Network Connection (`:n`)

```ic10
l r0 d0:0 Channel0   # read Channel0 from d0's connection 0 (data port)
s db:1 Channel0 r0   # write to Channel0 on db's connection 1 (power network)
```

## 5. Program Structure

### Comments

```ic10
# everything after # is ignored
alias MyAlias r0  # inline comment
```

### Labels

Labels mark line numbers for jump/branch targets:

```ic10
start:            # defines label "start" at this line number
j start           # jumps to label "start"
```

**Warning:** Do not name labels after IC10 keywords (`Temperature`, `Setting`, etc.) — this overwrites the keyword.

### Aliases

```ic10
alias vTemp r0       # "vTemp" now refers to r0
alias Sensor d0      # "Sensor" now refers to d0
```

Device aliases also label the corresponding screw on the IC Housing.

### Constants

```ic10
define pi 3.14159    # compile-time text substitution
move r0 pi           # equivalent to: move r0 3.14159
```

## 6. Stack Memory

Each IC10 has a **512-slot stack** (indices 0–511). Stack memory is **persistent** on the chip (survives code changes, but not chip duplication).

| Instruction | Effect |
| --- | --- |
| `push a(r?\|num)` | Write the value at index `sp`, then `sp++` |
| `pop r?` | `sp--`, then read stack[sp] into register |
| `peek r?` | Read stack[sp-1] into register (no sp change) |
| `poke addr(r?\|num) val(r?\|num)` | Write value to stack address (random access) |
| `clr d?` | Clear the stack of device `d?` |
| `clrd id(r?\|num)` | Clear stack of device by ReferenceId |
| `get r? (d?\|dr?) addr(r?\|num)` | Read stack value at address from device |
| `getd r? id(r?\|num) addr(r?\|num)` | Read stack value at address from device by ReferenceId |
| `put (d?\|dr?) addr(r?\|num) val(r?\|num)` | Write value to stack address on device |
| `putd id(r?\|num) addr(r?\|num) val(r?\|num)` | Write value to stack address on device by ReferenceId |

**Bounds:** `pop`/`peek` require sp ∈ [1, 512]; `push` requires sp ∈ [0, 511].

## 7. Device I/O

### Direct Device Access

| Instruction | Syntax | Description |
| --- | --- | --- |
| **l** | `l r? (d?\|dr?) logicType` | Load a LogicType value from a device into a register |
| **s** | `s (d?\|dr?) logicType r?` | Store a register value to a device LogicType |
| **ls** | `ls r? (d?\|dr?) slotIndex logicSlotType` | Load a slot property from a device |
| **ss** | `ss (d?\|dr?) slotIndex logicSlotType r?` | Store to a device slot |
| **lr** | `lr r? (d?\|dr?) reagentMode hash` | Load reagent info (Contents=0, Required=1, Recipe=2) |
| **rmap** | `rmap r? (d?\|dr?) reagentHash(r?\|num)` | Map reagent hash to the prefab hash the device expects |

### Direct Reference Access (by ReferenceId)

| Instruction | Syntax | Description |
| --- | --- | --- |
| **ld** | `ld r? id(r?\|num) logicType` | Load LogicType from device by ReferenceId |
| **sd** | `sd id(r?\|num) logicType r?` | Store to device by ReferenceId |

### Batch Access (all matching devices on network)

Batch instructions target all devices of a given **PrefabHash** (device type). Hashes are CRC-32 of the prefab name.

| Instruction | Syntax | Description |
| --- | --- | --- |
| **lb** | `lb r? deviceHash logicType batchMode` | Load from all matching devices |
| **sb** | `sb deviceHash logicType r?` | Store to all matching devices |
| **lbn** | `lbn r? deviceHash nameHash logicType batchMode` | Load filtered by name hash |
| **sbn** | `sbn deviceHash nameHash logicType r?` | Store filtered by name hash |
| **lbs** | `lbs r? deviceHash slotIndex logicSlotType batchMode` | Load slot from all matching |
| **sbs** | `sbs deviceHash slotIndex logicSlotType r?` | Store slot to all matching |
| **lbns** | `lbns r? deviceHash nameHash slotIndex logicSlotType batchMode` | Load slot filtered by name |

**Batch modes:**

| Mode | Value | No-match result |
| --- | --- | --- |
| Average | 0 | `nan` |
| Sum | 1 | `0` |
| Minimum | 2 | `0` *(was `pinf` in older versions)* |
| Maximum | 3 | `ninf` |

Either the word or the number can be used.

### Dynamic LogicType

The LogicType parameter can be a register value (the integer enum value), enabling runtime-selected properties:

```ic10
l r0 d0 r1    # r1 holds the integer value of the desired LogicType
```

## 8. Network Channels

Cable networks have 8 channels (`Channel0`–`Channel7`), accessed via connection syntax:

```ic10
l r0 d0:0 Channel3    # read Channel3 from d0's connection 0
s d0:0 Channel3 r0    # write to Channel3
```

Channels are **volatile** — destroyed when the cable network changes or the world is exited. Default value: quiet `NaN`.

### 9.4 Bitwise

All bitwise operations treat values as 64-bit signed two's complement integers.

| Instruction | Syntax | Description |
| --- | --- | --- |
| `and` | `and r? a(r?\|num) b(r?\|num)` | Bitwise AND |
| `or` | `or r? a(r?\|num) b(r?\|num)` | Bitwise OR |
| `xor` | `xor r? a(r?\|num) b(r?\|num)` | Bitwise XOR |
| `nor` | `nor r? a(r?\|num) b(r?\|num)` | Bitwise NOR |
| `not` | `not r? a(r?\|num)` | Bitwise NOT (complement). **Note:** `not 1` = `-2`, not `0`. Use `seqz` for logical NOT. |
| `sll` | `sll r? a(r?\|num) b(r?\|num)` | Logical left shift |
| `sla` | `sla r? a(r?\|num) b(r?\|num)` | Arithmetic left shift (identical to `sll`) |
| `srl` | `srl r? a(r?\|num) b(r?\|num)` | Logical right shift (zero-fill) |
| `sra` | `sra r? a(r?\|num) b(r?\|num)` | Arithmetic right shift (sign-extend) |
| `ext` | `ext r? src(r?\|num) offset(r?\|num) len(r?\|num)` | Extract bit field (max 53 bits) |
| `ins` | `ins r? field(r?\|num) offset(r?\|num) len(r?\|num)` | Insert bit field (max 53 bits). **⚠ BUG (stable as of 2026-01-12):** parameter order is `offset, len, field` in stable; `field, offset, len` in beta. |

**Missing gates:** XNOR and NAND have no dedicated instructions. Logical equivalents:

| Logic Gate | IC10 Equivalent |
| --- | --- |
| Logical NOT | `seqz r0 r1` |
| Logical AND | `min r0 r1 r2` |
| Logical OR | `max r0 r1 r2` |
| Logical XOR | `sne r0 r1 r2` (binary inputs only) |

### 9.5 Comparison / Set

All comparison instructions store **1** (true) or **0** (false) in the destination register.

| Instruction | Syntax | Condition |
| --- | --- | --- |
| `seq` | `seq r? a(r?\|num) b(r?\|num)` | a == b |
| `seqz` | `seqz r? a(r?\|num)` | a == 0 |
| `sne` | `sne r? a(r?\|num) b(r?\|num)` | a != b |
| `snez` | `snez r? a(r?\|num)` | a != 0 |
| `sgt` | `sgt r? a(r?\|num) b(r?\|num)` | a > b |
| `sgtz` | `sgtz r? a(r?\|num)` | a > 0 |
| `sge` | `sge r? a(r?\|num) b(r?\|num)` | a >= b |
| `sgez` | `sgez r? a(r?\|num)` | a >= 0 |
| `slt` | `slt r? a(r?\|num) b(r?\|num)` | a < b |
| `sltz` | `sltz r? a(r?\|num)` | a < 0 |
| `sle` | `sle r? a(r?\|num) b(r?\|num)` | a <= b |
| `slez` | `slez r? a(r?\|num)` | a <= 0 |
| `sap` | `sap r? a(r?\|num) b(r?\|num) c(r?\|num)` | approximately equal: `abs(a-b) <= max(c * max(abs(a), abs(b)), ε*8)` |
| `sapz` | `sapz r? a(r?\|num) b(r?\|num)` | approximately zero: `abs(a) <= max(b * abs(a), ε*8)` |
| `sna` | `sna r? a(r?\|num) b(r?\|num) c(r?\|num)` | not approximately equal (inverse of `sap`) |
| `snaz` | `snaz r? a(r?\|num) b(r?\|num)` | not approximately zero (inverse of `sapz`) |
| `snan` | `snan r? a(r?\|num)` | a is NaN |
| `snanz` | `snanz r? a(r?\|num)` | a is not NaN |
| `sdse` | `sdse r? (d?\|dr?)` | device is set (connected) |
| `sdns` | `sdns r? (d?\|dr?)` | device is not set |
| `select` | `select r? a(r?\|num) b(r?\|num) c(r?\|num)` | `r? = (a != 0) ? b : c` (ternary) |

### 9.6 Branching & Jumps

#### Unconditional

| Instruction | Syntax | Description |
| --- | --- | --- |
| `j` | `j target` | Jump to line/label |
| `jr` | `jr offset` | Relative jump |
| `jal` | `jal target` | Jump and store next line in `ra` |

#### Conditional Branching Pattern

Branches follow a systematic naming convention:

| Prefix | Meaning |
| --- | --- |
| `b-` | Branch to absolute line |
| `br-` | Branch relative |
| `b-al` | Branch and link (store return address in `ra`) |

| Suffix | Condition | Branch | Branch+Link | Relative |
| --- | --- | --- | --- | --- |
| `-eq a b` | a == b | `beq` | `beqal` | `breq` |
| `-eqz a` | a == 0 | `beqz` | `beqzal` | `breqz` |
| `-ne a b` | a != b | `bne` | `bneal` | `brne` |
| `-nez a` | a != 0 | `bnez` | `bnezal` | `brnez` |
| `-gt a b` | a > b | `bgt` | `bgtal` | `brgt` |
| `-gtz a` | a > 0 | `bgtz` | `bgtzal` | `brgtz` |
| `-ge a b` | a >= b | `bge` | `bgeal` | `brge` |
| `-gez a` | a >= 0 | `bgez` | `bgezal` | `brgez` |
| `-lt a b` | a < b | `blt` | `bltal` | `brlt` |
| `-ltz a` | a < 0 | `bltz` | `bltzal` | `brltz` |
| `-le a b` | a <= b | `ble` | `bleal` | `brle` |
| `-lez a` | a <= 0 | `blez` | `blezal` | `brlez` |
| `-ap a b c` | ≈ equal | `bap` | `bapal` | `brap` |
| `-apz a b` | ≈ zero | `bapz` | `bapzal` | `brapz` |
| `-na a b c` | ≉ equal | `bna` | `bnaal` | `brna` |
| `-naz a b` | ≉ zero | `bnaz` | `bnazal` | `brnaz` |
| `-nan a` | a is NaN | `bnan` | — | `brnan` |

#### Device Pin Branching

| Instruction | Syntax | Description |
| --- | --- | --- |
| `bdse` | `bdse d? target(r?\|num)` | Branch if device is set |
| `bdns` | `bdns d? target(r?\|num)` | Branch if device is not set |
| `bdseal` | `bdseal d? target(r?\|num)` | Branch+link if device is set |
| `bdnsal` | `bdnsal d? target(r?\|num)` | Branch+link if device is not set |
| `brdse` | `brdse d? offset(r?\|num)` | Relative branch if device is set |
| `brdns` | `brdns d? offset(r?\|num)` | Relative branch if device is not set |
| `bdnvl` | `bdnvl (d?\|dr?) logicType target(r?\|num)` | Branch if device invalid for load of logicType |
| `bdnvs` | `bdnvs (d?\|dr?) logicType target(r?\|num)` | Branch if device invalid for store of logicType |

**Branch target:** The last argument is always the target (absolute line, label, or relative offset) and accepts `(r?\|num)`. All comparison operands (`a`, `b`, `c`) also accept `(r?\|num)`. For `br-` relative branches, `breq a b 3` means "if a==b, jump 3 lines forward."

## 10. Subroutine Calling Convention

```ic10
jal myFunction       # saves next line number in ra, jumps to myFunction
# ...execution returns here...

myFunction:
  # ...do work...
  j ra               # return to caller
```

For nested calls, push/pop `ra` on the stack:

```ic10
outerFunction:
  push ra            # save return address
  jal innerFunction  # overwrites ra
  pop ra             # restore return address
  j ra               # return to outer caller
```

Maximum call depth using this pattern: 512 (stack size).

## 13. Ambiguities & Known Issues

| Issue | Details |
| --- | --- |
| **Float precision** | Wiki marks double-precision FP as "confirmed?" — likely correct but unconfirmed by developers. |
| **`mod` semantics** | Documented as "NOT `a % b`" — this is true mathematical modulo, not C-style remainder. `mod r0 360 -90` = 270. |
| **`ins` parameter order bug** | Stable build (as of 2026-01-12) uses `offset, length, field` order instead of documented `field, offset, length`. Beta has the correct order. |
| **Batch Minimum no-match** | Returns `0` as of v0.2.6091; was `pinf` in earlier versions. |
| **`bnaal` description** | Wiki says "branch if a != b" but the instruction name suggests "branch approximately and link" — likely a wiki error; treat as the `-ap` + `al` variant. |
| **Lines per tick** | Experimentally ~128, but the exact count shows a +129/+129/+126 pattern, suggesting 128 is the intended value with minor counting artifacts. |
| **`sleep` precision** | Pauses for `a` seconds, but precision relative to tick boundaries is undocumented. |
| **`clr db` in non-housing** | Using `clr db` when the IC is not in an IC Housing (e.g., in an Air Conditioner slot) causes an error. |
| **Hexadecimal limit** | Values > 54 bits in hex literals "might get corrupted" — unclear exact threshold. |
